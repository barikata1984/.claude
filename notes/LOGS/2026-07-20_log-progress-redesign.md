# 2026-07-20 log-progress-redesign

## Topic

log-progress スキルの全面再設計。「ログを後から読んでも、なんでそうなったかが分からない」という不満の解消が目的。要件定義から実装、4 回の評価反復、トリガー検証、資産整理までを行った。

## History

出発点はユーザーの不満だった。既存の log-progress はトピック別ログ (`log_*.md`) への追記方式で、決定の背景や却下案が残らず、記録として機能していない。ユーザーは compact-plus (コンパクト時の状態引き継ぎプラグイン) を発見し、これを進捗記録用に転用する案を持ち込んだ。

批評の結果、転用案の半分を採用し半分を退けた。compact-plus の Session Decisions / Failed Attempts (根拠・却下案・失敗理由の記録) は不満の根因に直撃するため移植する。一方で本体は復帰エージェント向けの状態スナップショットで、Worker Topology などの揮発セクションは議事録に不要。TODO/ISSUES の廃止案は、イベントログ (議事録) と状態インデックスの混同として退けた (状態の投影がないと、残タスクの把握に全記録の再生が必要になる)。

要件定義は grilling スキル (mattpocock/skills から導入) の 1 問 1 答形式で行った。記録の一次単位はセッション議事録への一本化とし、トピック別ログとの併存案は二重記帳になり必ず片方が腐るため却下した。発動は手動 + wrap-up-session 連動の 2 経路にとどめ、PreCompact フックの導入は hooks 設定と発動タイミングのデバッグまでスコープが膨らむため見送った (必要になれば compact-plus 自体を入れれば補完関係になる)。死んでいると確認された daily-report は削除した。旧構成前提のまま延命・再構築するより、必要になったとき議事録ベースで作り直す方が安い。既存の log_*.md は凍結残置とした。ARCHIVE 移動は既存の参照リンクを切り、議事録形式への変換は元のセッション文脈が既に存在しないため疑似議事録の捏造になる。

書き手の決定が要だった。「サブに書かせてメインが検証」は検証の向きが逆 (会話の全文脈を持つのはメインだけで、要約段階で落ちた情報は復元不能) という理由から、メイン直書き + サブ lint に反転させた。

初版はフル固定 7 セクション (Context / Decisions / Failed Attempts / Findings / Changes / Open Items) で実装し、旧版とのベンチマーク (3 シナリオ、機械 assertion) で新 100% vs 旧 68.3% を得た。ただしテストハーネスは完全な経過をプロンプトで渡すため実運用の要約損失は再現されず、差は構造系 assertion のみに由来する。

ユーザーレビューが設計の転回点になった。7 セクション版の出力には「案 B が案 A より先に現れ、何を言っているか分からない」という指摘が入った。問題解決の因果 (問題 → 仮説 → 検証 → 判断) がカテゴリ別セクションに分断されていたためで、原因は compact-plus のフォーマットが状態スナップショット (カテゴリ引き) であり、人間が読む物語と構造が異なることにある。セクションの順序を入れ替えるだけの案も検討したが、一つの因果連鎖が複数セクションに割れる分断自体は解消されないため退け、テンプレートを物語中心に再設計した:

- セクションは Topic / History / Decisions / Changes / Open Items の 5 つ
- History が因果順の散文で理由・却下案・失敗を運ぶ
- Decisions は 1 行索引 (採用案 / 却下: 案名 — 承認区分)。根拠は書かない (History が運ぶ)
- プロジェクト単位の決定台帳 DECISIONS.md を新設 (ADR の軽量版)。台帳を持たず散文への grep に頼る案は、議事録が蓄積するほど検索対象が散らばり劣化するうえ、決定はタスクでも未解決課題でもなく TODO/ISSUES に投影されないため却下。議事録の Decisions 行と同一テキストで同期し、lint が一致を検査

物語セクションの見出しは、Timeline (時刻表を予告)、Sequence of Events (複数単語で他の見出しから浮く)、Narrative (語る行為が前景に出る)、Background と Circumstances (静的前提を予告)、Process (手順を予告)、What happened (名詞句の並びに文が浮く)、経緯 (見出し言語の不統一) を検討・却下した末、History に確定した。

再評価 (新テンプレート vs 7 セクション版) は 100% vs 59.7%。lint の逆方向検証が実際に機能し、結論先出しの段落や議事録と台帳の文言不一致を検出・修正させ、根拠のない数値の捏造要求は正しく拒否された。

続くフィードバックで、並列項目 (競合する仮説群・候補案) を箇条書きで導入し散文で 1 つずつ解決する規則を追加した。従来の散文埋め込みでは選択肢の集合が一望できず、「案 B を見た瞬間に案 A はどこかと探す」読みにくさが項目列挙の粒度で再発するためで、埋め込み継続の却下理由もこれである。検証 run では列挙形が全 3 ケースで再現された。

次のレビューで「判断と実装が 1 段落に詰まる」という指摘を受けた。原因を調査したところ、列挙規則が段落の単位を「決着の回数」に暗黙に再定義した副作用と確定した (規則追加前の出力では判断と実装は別段落だった。介入変数は規則追加のみ)。対策として、段落は作業の単位 (仮説の検証 / 原因の特定 / 案の比較と判断 / 実装と検証) で切り、短い作業同士の結合は許す規則を追加し、最終検証で全 11 assertion をパスした。

description のトリガー検証では、skill-creator 標準の run_loop が三重に破綻した。セッションリミット中のエラーが全件「非発動」として集計され、search-first フックにより最初のツールが検索になると即・非発動判定され、候補 description を持つスタブが本物のスキルに遮蔽されて呼ばれない。さらに本質的な限界として、`claude -p` のセッションは空であり、記録スキルには正当に「書くことがない」。

そこでパッチ測定器を自作した: 本物のスキル名の呼び出しを発動として数え、ストリーム全体を走査し、全クエリに一様な作業前置きを付す。sonnet・10 クエリ (18 から削減、ユーザーレビュー済み) の測定結果は precision 5/5 (「コミットしてプッシュ」は commit-and-push へ、「log_sweep.md に記録」は sweep-analyze へと正しくルーティング)、recall 3/5 確実 + 2 件が不安定。不安定 2 件はモデルが「会話に実体がなく書くと捏造になる」と手を止めたもので、スキル自身が要求する誠実さが原因であり実セッションでは成立しない失敗様式のため、description は現状維持とし最適化ループは回さなかった (完璧な precision 側を崩す過学習リスク)。

評価資産は `skills/log-progress/evals/` へ移設した (回帰評価 3 シナリオ + fixtures、トリガー測定器 + 10 クエリ、イテレーション履歴の要約)。作業場ディレクトリは削除。副産物として、この環境の zsh は `=` で始まる語をコマンド検索するため `echo ====` 等が `(eval):1: == not found` で落ちることを特定した (標準ハーネスのクラッシュ原因の一つ)。

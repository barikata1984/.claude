# Log: request-source

## 2026-06-05: request-source スキル全面改訂 + Sources ブロック自動付与の導入

### 動機
ユーザーが「応答末尾に常に情報ソースを書き込むようにしたい」と要望。目的は内部知識のみに依存した回答をユーザーが検出できるようにすること。

### 設計議論（採用前の批判的検討）
ユーザーの初期案（Internal/External 分類で全応答に付与）に対し 4 点を指摘:
1. 全応答に付けるのは過剰（コード編集・会話的応答にはノイズ）
2. Internal の列挙は本質的に不正確（訓練データは書誌ではない）
3. 既存の /request-source スキルと重複
4. 事後ラベルより事前の検証強制が効く（CLAUDE.md の外部事実の検証ルールがそれ）

合意した方針: Internal/External → [検証済]/[未検証] の二値分類、外部事実を含む応答にのみ付与、既存 /request-source を改修して一本化。

### 実装
- **SKILL.md 全面書き換え**: 旧 3 分類（内部知識/外部知識/観測事実）を [検証済]/[未検証] 二値に変更。自動付与モード（応答末尾に `---` + Sources ブロック）と手動モード（`Sources:` から開始、`---` なし）の 2 モード。
- **CLAUDE.md 追記**: 「外部事実の検証」セクション末尾に自動付与ルール 1 行追加。

### empirical-prompt-tuning（5 iter + hold-out、収束）

シナリオ:
- A（中央値）: PyTorch torch.compile の技術質問（WebSearch + 内部知識混在）
- B（edge 負例）: リファクタリング実行報告（外部事実なし）
- C（edge 手動）: 「ソースは?」による遡及注釈
- D（hold-out）: Claude 4 の新機能についてのニュース系質問

修正履歴:

| iter | 修正テーマ | 効果 |
|---|---|---|
| 1 | (baseline) | 全シナリオ 100%. 手動モードの `---` 問題、粒度曖昧を検出 |
| 2 | 手動モード `---` 不要を明記 + 粒度ガイドライン追加 | C の不明瞭点解消. CLAUDE.md 二重記載問題を検出 |
| 3 | inline 注記と Sources の責務分離セクション追加 | 二重記載解消. 「誰にとって自明か」を検出 |
| 4 | 「実務者にとって自明」に限定 + hold-out 追加 | A 収束(2 連続ゼロ). hold-out で常識/製品事実の境界を検出 |
| 5 | 「学部教育を受けた者」に変更(ユーザー FB) + 粒度/分類の優先順位明記 | A 2 連続ゼロ確認. hold-out 精度 100%(過適合なし) |

SKILL.md への変更は計 5 箇所:
1. 手動モードで `---` 不要、`Sources:` から始めると明記
2. 粒度: 同一ソースの密接な事実はまとめてよい
3. Sources ブロック付与時は本文中の inline 注記(`(内部知識・未確認)` 等)を省略（CLAUDE.md との責務分離）
4. 「自明な常識」の基準を「当該分野の学部教育を受けた者」に設定（ユーザー FB）
5. 粒度ルール(除外) > 分類ルール(記載)の優先順位を明記

### その他の変更

## 2026-06-08: request-source [未検証] マーク漏れ対策 + 用語体系整理

### 問題
ユーザー報告: 内部知識で参照情報を述べたとき, 応答末尾の Sources ブロックに `[未検証]` マークが付かないケースが多い.

### 原因分析（EPT Iteration 0）
SKILL.md の構造的問題を 2 点特定:
1. 分類ルールが [検証済] と [未検証] を対称的に定義しているだけで, 「検証しなかった参照情報は必ず列挙せよ」という積極的指示がない. モデルは WebSearch を行った事実（行動の記憶）は想起しやすいが, 行わなかった事実を自己申告する動機が弱い.
2. CLAUDE.md 連携セクションが本文中の `(内部知識・未確認)` を Sources ブロックに集約して省略する設計のため, Sources で [未検証] が漏れるとどちらにもマークされない盲点が生まれる.

### EPT 検証（3 iter + hold-out, 収束）

シナリオ:
- A（中央値）: asyncio.gather() の API 仕様を内部知識で回答
- B（edge）: pytest 設定（観測情報）+ pytest 8.x 新機能（内部知識）の混在
- C（hold-out）: Attention Is All You Need の論文情報を内部知識で回答

修正 2 件:
1. **核心原則セクション追加**: 参照情報を「デフォルトで [未検証]」と位置づけ, WebSearch 等で確認したものだけ [検証済] に昇格する原則を明示. 応答完成後に各事実主張を走査する自己チェック手順を追加.
2. **CLAUDE.md フォールバック安全弁**: Sources ブロックを付与しない応答では CLAUDE.md のルールに従い本文中に「内部知識・未確認」を明示するフォールバックを明記. 「どちらか一方は必ず適用. 両方とも適用しない状態は許されない」.

全 iter で全シナリオ精度 100%. EPT テスト条件ではスキルを明示的に読ませるためバックグラウンドコンテキスト落ちは再現できない制約あり.

### 用語体系の整理

ユーザーから「外部事実」という複合語がしっくりこないとのフィードバック. 「外部」が何に対して外部なのか曖昧で, 「外部知識」（セッション外 = WebSearch 等）と混同しやすい.

変更:
- **外部事実 → 参照情報**: 出典を示せるべき知識. Sources ブロックの対象
- **セッション内観測事実 → 観測情報**: セッション中にツールで得たもの. Sources の対象外

適用箇所: SKILL.md（8 箇所）+ CLAUDE.md（4 箇所）

用語体系は 2 軸で整理:
- 情報種別の軸: 参照情報 / 観測情報
- 知識源の軸: 内部知識（[未検証]）/ 検証済み（[検証済]）

用語変更後の品質検証（3 シナリオ）で劣化なしを確認.

### 残存課題
- 「学部教育レベルで自明な事実」の境界が全 iter で繰り返し指摘されたが, [未検証] 漏れの原因ではないため修正対象外
- 実運用でのバックグラウンドコンテキスト落ちは EPT では測定不能. 実セッションでの観察待ち


## 2026-06-08（続）: request-source「自明な常識」の境界を操作的に定義

### 問題
前回 EPT（2026-06-08）で「学部教育レベルで自明な事実」の境界が全 iter で繰り返し指摘されていたが, [未検証] 漏れの直接原因ではないため修正対象外としていた. 今回, この曖昧さ自体を EPT で解消.

### EPT 検証（3 iter + hold-out, 収束）

シナリオ:
- A（中央値）: React/Next.js の技術説明（基礎〜バージョン固有情報が混在）
- B（エッジ）: ML モデルの K8s デプロイ（複数分野の知識が交差）
- C（hold-out）: DB 選定助言（PostgreSQL/MongoDB 比較, 別ドメインでの汎化検証）

| iter | テーマ | A retries | B retries | A 精度 | B 精度 |
|---|---|---|---|---|---|
| 1（ベースライン）| — | 1 | 1 | 100% | 100% |
| 2 | 除外/含める基準 + 迷ったら含める | 1 | 0 | 100% | 100% |
| 3 | 除外基準を「プロダクト非依存」に限定 | 1 | 0 | 100% | 100% |
| hold-out | — | — | — | 100%（C）| — |

### Iter 1 で検出された不明瞭点
- 「当該分野の学部教育を受けた者にとって自明な常識」の「当該分野」が未定義（複数分野にまたがる応答で基準不明）
- React の仮想 DOM, REST API 慣行等の borderline ケースで両 agent とも 1 回やり直し

### 適用した修正（2 テーマ, 記載の粒度セクション）

1. **除外/含める基準の操作的定義 + 「迷ったら含める」デフォルト**:
   - 旧: 「当該分野の学部教育を受けた者にとって自明な常識」（曖昧）
   - 新: 除外基準（プロダクト非依存の分野共通概念に限定, 例示付き）, 含める基準（プロダクト固有, 慣行判断, 数値）, 迷ったら含める（除外の過誤 > 冗長の過誤）
   - 効果: B の retries 1→0, B の不明瞭点 2→0

2. **除外基準に「プロダクト固有特性は除外しない」を明記**:
   - 旧: 除外基準の例示のみ
   - 新: 「特定のライブラリ・フレームワーク・ツールの特性は, たとえ広く知られていても除外しない」
   - 効果: A の不明瞭点 1→0（ただし retry 1 は構造的に残存: 初期直感「除外」→明示ルールで修正, 安全弁として機能）

### Hold-out 結果
DB 選定シナリオ（C）: 全 6 要件クリア, retries 0, 精度 100%. 別ドメインへの汎化に問題なし, 過適合なし.

### 残存
- A の retry 1 は構造的（「入門教科書の概念」と「特定プロダクトの事実」の重なりで初期直感が揺れる）. 最終判定は常に正しいため, 安全弁として許容.

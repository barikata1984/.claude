## 2026-05-30（続々）: under-retrieval 対策「外部事実の検証」節を新設（段階 1）

### 問題の切り分け（hallucination ではなく under-retrieval）
調査・デバッグ依頼で、(a) 検索すべき外部知識を取りに行かず内部知識だけで答える、(b) 検索しても浅く取り残す、という困りごと。当初 hallucination 枠で調査を始めたが、議論の結果 under-retrieval（外部知識導入のサボり = 行動選択の失敗）と切り分けた。hallucination（生成の捏造）はその下流の二次的帰結であって別物。対策の重心は「検索を起動・深化させる」に置き、confidence 宣言・claim 撤回等の hallucination 対策一式は不要と判断。

### 批判的スタンスとの分離
当初 CLAUDE.md の批判的スタンス節に相乗りさせようとしたが、規律のカテゴリが別（批判的スタンス = 対ユーザーの態度 / 外部事実の検証 = 対・事実の情報取得衛生）。混ぜると節の凝集性が壊れ、empirical でも両者の A/B が干渉するため、独立節として新設。共通の上流は karpathy §1。

### 既存実装の調査
- Anthropic 公式「Reduce hallucinations」: 「I don't know 許可」「claim 撤回」「外部知識制限」。
- Aedelon/claude-code-anti-hallucination: Haiku 4.5 が prompt を VOLATILE/NONE に二値分類し、VOLATILE のときだけ UserPromptSubmit hook で検証プロトコルを inline 注入（NONE は素通り）。自己申告 F1 = 0.988。「ゲート（分類）とプロトコル（実行）の責務分離」が最大の収穫 = over-trigger 回避の鍵。
- Aedelon/claude-code-blueprint: research-protocol skill（SCOPE → SEARCH → EVALUATE → SYNTHESIZE → CITE → ASSESS、source hierarchy）。失敗 (b) の深さ対策は既存 deep-research でカバー可。

### 設計判断
- 強制はハーネスでは原理的に不可能（検索する/しないの判断はモデル内部で起き、PreToolUse 等では観測不可）。
- Stop hook の事後ゲート（検索ゼロでの終了を差し戻す）は却下: スクリプトは意味判定できず、全質問に検索を強要して over-response を招く（CLAUDE.md の抑制方針と衝突）。
- 常駐 Haiku デーモンは個人環境に過剰・MCP 依存・F1 が自己申告のため不採用。思想（分類 → 注入の責務分離）のみ借用。
- 段階導入を採用: 段階 1 = CLAUDE.md に薄い指示節、段階 2 = empirical 測定、段階 3 = 忘却が残れば軽量 UserPromptSubmit hook（正規表現分類）。デーモン・hallucination 対策一式は持ち込まない。

### 段階 1 の実装
CLAUDE.md に新節「外部事実の検証」を追加（実験コメント付き）。トリガーを「外部事実を述べるとき」に限定（内部知識で確実なもの・trivial は素通り）、確認できなければ「内部知識・未確認」と明示、深さの要る調査は deep-research に委譲。

### empirical-prompt-tuning（収束）
5 シナリオ、いずれも新規 subagent に節を渡して挙動を観察:
- A（バージョン固有 API、検索すべき）: WebSearch/WebFetch 起動・出典明示・未確認明示。○
- B（再帰関数、一般概念）: 検索ゼロで即答。○
- C（git --amend）/ D（HTTP 403/404）境界: "仕様" に過剰反応せず検索ゼロ。over-trigger なし。○
- E（Node.js 最新 LTS、hold-out / under-trigger 境界）: 検索起動、出典の矛盾を一次情報源で裁定。○
全シナリオ意図通り（検索すべき場面で起動、不要な場面で抑制）。新規不明瞭点が連続ゼロ・精度飽和・過適合なしで収束。文面は修正不要だった。

### 限界と残課題
- empirical は「指示の明瞭さ（白紙の実行者が場合分けできるか）」を測れたが、「実運用の遵守率（CLAUDE.md に書いてあるのにコスト判断で従わないか）」は測れない。これが最大リスクで、実運用観察待ち。
- 全シナリオで subagent が「補足を足すか、抑制節と緊張する」と裁量補完で迷った = 外部事実の検証節ではなく批判的スタンスの抑制節との相互作用。別途観察。

## 2026-06-08: 参照情報の検証節を段階 1 → 段階 3 に強化（EPT 5 iter + hold-out）

### 動機
ユーザー報告: 技術質問・論文ディスカッション・STEM 系の議論で, Claude がまず内部知識で答えようとし, 外部情報を参照すべき場面で under-retrieval が発生. 段階 1 の指示（「参照情報を述べるときは検証」）が出力時トリガーのため, 回答構成後の追認検索になり構造的に防げない.

### 方針決定: skill-creator vs EPT
EPT を選択. 理由: (1) 新機能ではなく既存指示の効力の問題, (2) skill にするとトリガーが広すぎ他 skill と競合, (3) EPT の設計意図に直接合致.

### Iteration 0（静的整合チェック）
- セクション名「参照情報の**検証**」= 確認・追認のフレーミング
- HTML コメントの目標「under-retrieval 対策」= 検索頻度を上げたい
- 本文トリガー「参照情報を**述べるとき**は」= 出力時制約
- 乖離: 目標は proactive retrieval, 指示は reactive verification

### EPT シナリオ（全 iter 共通, 3 種 + hold-out 1 種）
- A（中央値）: PyTorch `torch.compile` の mode パラメータ（API 仕様）
- B（edge）: MAE 論文の ViT-Large ImageNet fine-tuning Top-1 精度（論文数値）
- C（edge）: LayerNorm vs BatchNorm の数式 + `nn.LayerNorm` の `elementwise_affine`（数式+API）
- D（hold-out）: Python `asyncio.TaskGroup` の導入バージョンと `gather` との違い

### メトリクス推移

| iter | 平均精度 | 平均 steps | 平均 duration | 新規不明瞭点 |

精度は全 iter 100%（天井効果, 要件チェックリストの行動誘導による）. 改善は質的指標（不明瞭点の解消）で追跡.

## 2026-06-19: 検索優先ルールの顕著性強化（CLAUDE.md 改訂 + フック導入）

### 問題の再定義: 理解失敗ではなく信頼度バイアス

EPT iter 1 で全シナリオ 100% を確認済みの段階で, 実運用での under-retrieval が継続. 原因を調査した結果, ルール文面の曖昧さではなく「内部知識に確信が持てるとき検索をスキップする」という信頼度バイアス (confidence bias) と特定. ルールの comprehension は問題なく, salience（運用時の発火率）が実問題.

### CLAUDE.md 改訂

カテゴリ列挙方式（API・ライブラリ・ハードウェア等の列挙）から「技術的・科学的な問いをすべて対象」とする包括的ルールに書き換え. 信頼度に関係なく検索を最初の行動とする旨を明示した.

### EPT（iter 1）

3 シナリオすべて 100%: ルール文面はすでに十分明瞭で, 修正の余地なし. 問題が salience にあることを確認.

### 顕著性対策: フック導入

CLAUDE.md の指示単体では salience が不足と判断し, Claude Code フックを追加:

- **UserPromptSubmit フック** (`~/.claude/hooks/search-first-reminder.sh`): 毎ターン, ユーザープロンプト受信時に検索優先リマインダーを `system_prompt` として注入する. これにより CLAUDE.md の指示がシステムプロンプトの末尾で毎回強調される.
- **SubagentStart フック** (`~/.claude/hooks/search-first-reminder-subagent.sh`): サブエージェント起動時にも同リマインダーを注入. サブエージェントは親の CLAUDE.md を引き継ぐが, システムプロンプト末尾への強調は継承されないため独立して追加.

両フックを `settings.json` の `hooks.UserPromptSubmit` / `hooks.SubagentStart` に登録済み.

### 残課題

- 実運用での効果観察（フック注入で under-retrieval が実際に減るか）
- フックが毎ターン発火することによるオーバーヘッドの許容評価

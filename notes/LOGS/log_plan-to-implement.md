# plan-to-implement スキル設計ログ

## 2026-07-02: 設計討議（Stage 1 起草前）

### 背景

新スキル `plan-to-implement`（`/implement` から呼ぶ）の設計討議を行った。実装（コード）はまだ書いていない。機能主導のインターフェース交渉によってエージェントチームの並列実装を駆動するオーケストレータとして構想し、設計判断とロードマップを固めた段階。

### 到達した設計判断

1. **ギャップの特定**: `~/.claude/rules/agent-teams.md` はチーム編成の機構（TaskCreate・依存・owner・shutdown）は定義するが、「チームで分割 vs 単一サブエージェント」のトリアージ判断を定義していない。この判断を埋めるのが目的。
2. **置き場所はスキル（ルールではない）**: 単独でも `/implement` からでも呼べる＝起動可能・合成可能であり、多フェーズの手続きは嵩がある。ルールは常時ロードの受動的コンテキストで呼び出せず合成できないため不適。方針の門番一行だけは将来 `agent-teams.md` に置きうる（条件付き）。
3. **トリアージゲートを2問に集約**: (a) 広さ方向の作業（独立した多数の対象）か深さ方向（密結合な単一物）か、(b) いま継ぎ目（seam＝機能間の境界の契約）を名指しできるか。広さ＋名指し可 → チーム、深さ or 名指し不可 → 単一。
4. **機能主導のインターフェース交渉**（自動車/プラントの ICD の喩え）: 各機能担当が I/O 要求を申告 → インターフェース担当が衝突を調停して契約案を提案（調停権を持つ）→ 各機能担当がレビュー → 全 OK まで ≤3 ラウンド反復 → 契約凍結。非収束なら単一降格・再分割・ユーザーへエスカレーション。
5. **messaging は計画フェーズに限定**（凍結契約を生む手段）。実装フェーズは凍結契約を読むだけで messaging なし。
6. **トポロジは星型**: インターフェース担当をハブに、機能担当はハブ経由で交渉。全対全（O(N²)）を避ける。大 N は ~4 群への再帰的メタ分割で捌く。
7. **非収束＝分解が間違っている信号**。トリアージそのものになる。
8. **価値の再定義**: 目的は第一に速度ではなく、単独では維持できないインターフェース調停の認知負荷の肩代わり。成功条件は「維持可能な契約が出たか」。

### ロードマップ（実装順）

- **最小版 v1 = Stage 1（交渉フロー）+ `/implement` ラッパ**。`/implement` 統合はユースケース上の目的そのものであるため最小版に含める（当初 Stage 5 から移動）。素直なタスクなら完全版と同等の成果。
- **完全版に必須（critical）= Stage 2**: トリアージと非収束処理の硬化（単一 vs チーム判定精度、デッドロック時のエスカレーション、継ぎ目名指しの粒度）。ept の指摘が主にここに出る想定。
- **条件付き（必要が観測されたら）= Stage 3（大 N の再帰的メタ分割）・Stage 4（実装中の契約再交渉）・`agent-teams.md` の方針一行**。
- 検証は各段階で ept（empirical-prompt-tuning）で二面評価し反復。

### 技術的確認事項（claude-code-guide エージェントで公式ドキュメントを確認）

- plan モードはメインセッションの permission mode で手動切り替え（Shift+Tab）。サブエージェント個別のモードではない。
- サブエージェントは subagent_type ごとの独立権限で動く。auto モードではサブエージェントの `permissionMode` 指定は無視される。
- read-only 探索のイディオムは「起動時にモード切り替え」ではなく **read-only の agent type（Explore）を選ぶ**こと。
- 設計含意: Stage 1 の探索 = メインが Explore サブエージェントを spawn → 調査結果を受け取り機能分解を決定 → 機能担当を spawn。
- 未確認: auto モード以外での subagent frontmatter `permissionMode: plan` の可否、read-only 探索の公式ベストプラクティスと Explore 型の正確な定義。

- 未定義の略語「MVP」を展開せず使い指摘された。既存用語も定義なしに使わない。

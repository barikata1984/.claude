# サブエージェント役割分担・モデルルーティング

## 2026-07-04: 経緯と設計

別セッションでの試行錯誤から、以下の運用パターンが使い心地良いと判明:
- メインループ（対話）は Sonnet
- 調査・実装・分析などの実作業はバックグラウンドのサブエージェントに委譲
- サブエージェントのモデルは役割で分ける: 調査・文書作成 = Sonnet, 実装・分析 = Opus
- メインループはサブエージェントの出力をレビュー・編纂してから報告

これを CLAUDE.md / エージェント定義に恒常化するため調査した。

### 確認した事実（公式ドキュメント）

- サブエージェントのモデル解決順位: ① `CLAUDE_CODE_SUBAGENT_MODEL` 環境変数 → ② Agent ツール呼び出し時の `model` パラメータ → ③ サブエージェント frontmatter の `model` → ④ メイン会話のモデル（[sub-agents](https://code.claude.com/docs/en/sub-agents.md), [model-config](https://code.claude.com/docs/en/model-config.md)）
- v2.1.198 以降、サブエージェントはデフォルトでバックグラウンド実行
- `~/.claude/rules/*.md` はユーザーレベルで全プロジェクトに自動読み込みされる（`.claude/CLAUDE.md` からの明示的 `@import` 不要、公式仕様）

### 現状確認で見つかった問題

- 観察されていた「実装・分析は Opus」を裏付ける設定はどこにも存在しなかった（登録済みカスタムエージェント `ideator`/`novelty-checker`/`red-teamer` は全て `model: sonnet`）。おそらくメインループ自身がその場の判断で `model` パラメータを渡していただけで、恒常的な設定ではなかった
- `agents/pt-engineer.md` / `pt-analyst.md` / `pt-research.md` は YAML frontmatter が存在せず、登録済みサブエージェントとして機能していなかった（`plan-to-implement`/`implement` スキルからも参照なし = 孤立ファイル）
- これらは 2026-03-22 の `project-team` スキル設計時に作られたもので、project-team スキル本体（フェーズ管理・ハンドオフ・状態ファイルを伴う重量級フレームワーク）は 2026-04-06 に無効化、2026-05-04 に `abff630` で完全削除されていた（理由の記録なし）

### 設計判断

project-team のような「スキル内に役割別エージェントを埋め込み、フェーズ/状態ファイルで管理する」設計は使われずに終わった。同じ轍を踏まないため、今回は以下の方針で最小構成にする:

- 役割別エージェント（engineer/analyst/researcher）はどのスキルにも属さない独立したプリミティブとして定義する。特定のワークフロー（project-team のような）に埋め込まない
- フェーズ管理・ハンドオフ・状態ファイル・Critique Mode は導入しない。ユーザーが観察した必要な要素は「モデル固定」と「報告前レビュー」の2点のみ
- `pt-engineer.md`/`pt-analyst.md`/`pt-research.md` の内容（実装/分析/調査タスクの具体的な作業内容・規約・出力フォーマット）は質が高く再利用可能と判断し、project-team 依存部分のみ除去して流用

### 実装内容

| ファイル | 内容 |
| ---- | ---- |
| `agents/engineer.md` | コード実装・テスト・インフラ担当、`model: opus`、tools を Read/Edit/Write/Bash/Grep/Glob/Agent に制限 |
| `agents/analyst.md` | 実験設計・統計分析担当、`model: opus`、tools を Read/Edit/Write/Bash/Grep/Glob に制限 |
| `agents/researcher.md` | 文献サーベイ・執筆担当、`model: sonnet`、tools を Read/Write/Edit/Grep/WebSearch/WebFetch に制限 |
| `rules/delegation.md` | 役割別ルーティング表（作業内容 → エージェント → モデル）+ 報告前レビュー方針 |

`presenter` のような追加ロールは、実際に必要になった時点で同じ型で追加する方針とし、今回は作らない（要求されていない機能を先回りしない）。

## 2026-07-04: `writer` エージェントの追加

ユーザーが別途確認した経験則: Opus（特に 4.8）はライティング能力が著しく低く、現行ハーネス + Sonnet の応答が文書作成では最も好ましい。この観察に基づき、文書執筆を専門とする `writer` エージェントを追加した。

- `agents/writer.md`（`model: sonnet`。他の3ロールと技術的には同じ frontmatter 指定であり、「固定」という特別な機構があるわけではない。当初その旨を強調する記述を加えたが、他ロールとの扱いの違いを誤って示唆するため削除した）
- `researcher.md` から「論文執筆」「提案書執筆」タスクと関連する description の文言を除去し、`writer` に委譲する形に整理（役割説明の重複はサブエージェント自動選択の精度を下げるため）
- `rules/delegation.md` のルーティング表を他ロールと同じ表記（モデル名のみ）に統一

### 既知の限界

- サブエージェントへの自動委譲判定（description に基づく）自体は Claude の裁量であり、ハーネス機能として強制はできない。今回固定できたのは「委譲された場合のモデル」のみ
- 「報告前に必ずレビューする」もプロンプト指示であり、強制メカニズムはない（CLAUDE.md 全般の制約）

## 2026-07-04: EPT iteration 1 の実施と「delegation.md 未ロード」の発見

上記設計（`agents/engineer.md`・`analyst.md`・`researcher.md`・`writer.md` と `rules/delegation.md`）に対し、empirical-prompt-tuning の手順で iteration 0（静的整合性チェック）・iteration 1（実行検証）を実施した。

### Iteration 0（静的チェック）

4エージェントの description と本文の整合性を確認。問題なし。

### Iteration 1（実行検証）

新規サブエージェントに対し、以下3シナリオを与えて挙動を観察した:

- シナリオA: 実装タスク（`engineer` への委譲を期待）
- シナリオB: 執筆タスク（`writer` への委譲を期待）
- シナリオC: 些細な1行修正（委譲すべきでない）

結果:

| シナリオ | 期待 | 実測精度 |
| --- | --- | --- |
| A（実装） | engineer に委譲 | 33% |
| B（執筆） | writer に委譲 | 63% |
| C（些細な修正） | 委譲しない | 100% |

シナリオA・Bはいずれも委譲されず、実行エージェント自身がタスクを自力完結させるケースが多数を占めた。シナリオCのみ意図通り「委譲を避ける」判断が一貫していた。

### 決定的な診断

上記結果の原因を特定するため、追加の診断サブエージェントに「delegation 関連の指示を保持しているか」を尋ねた。回答から、`rules/delegation.md` がロード済みコンテキストに一切含まれていないことが判明した。実際にロードされていたのは CLAUDE.md、karpathy CLAUDE.md、japanese-tech-writing.md、concise、agent-teams.md、workflow.md のみだった。

**原因**: `~/.claude/rules/*.md` はユーザーレベルで全プロジェクトに自動読み込みされるが、その読み込みは**セッション開始時に一度だけ**行われる静的スナップショットであり、動的な再スキャンは行われない。`delegation.md` は本セッションの作業中に新規作成したファイルであるため、本セッション中に生成されたメインループ自身にもどのサブエージェントにも反映されない。

これは対照的に、`agents/*.md`（カスタムサブエージェント定義）はセッション中も動的にスキャンされ、作成直後から `Agent` ツールの `subagent_type` として利用可能になる、という非対称性を実地で確認した結果でもある。

### 結論

今回の EPT 結果（33%/63%/100%）は `rules/delegation.md` の文言の実効性を測る有効なデータではない。文言そのものの評価は、`delegation.md` が静的スナップショットに含まれる次回セッション開始後、同じ3シナリオを再実行して初めて可能になる（TODO.md に申し送り済み）。

### 副産物: Write ツールの report.md ガード

シナリオB（執筆タスク）の実行中、Write ツールがファイル名 `report.md` を「サブエージェントは所見をテキストで返すべきで、報告ファイルを書くべきではない」というガードでブロックする挙動が観察された。ユーザーが明示的に指定した保存先パスであってもブロックされた。これは今回の委譲設計とは別の、既存ハーネスのガードの過剰適用であり、今回は対応せず ISSUES.md に記録するに留めた。

## 2026-07-04: Write ガード問題の深掘り調査と恒久対策

上記の副産物（Write ガードが `summary.md` のような正当な成果物パスもブロックする問題）について、原因調査・設計上の気づき・対策・再検証まで実施した。

### 調査

GitHub issues（anthropics/claude-code）、Reddit、Hacker News、公式ドキュメント（sub-agents.md, hooks-guide.md）を調査したが、この挙動についての公開報告は見当たらなかった。`~/.claude/settings.json` の hooks にも該当する PreToolUse ブロックは定義されていない（確認済み、PostToolUse のフォーマッタのみ）。ハーネスの明示的な hook 機構ではなく、モデル自身が「サブエージェントは所見をテキストで返すべき」という規範を内面化し、ファイル名パターン（report/summary 等）を見て自発的に Write を思いとどまっている可能性が高いと判断した。

### 設計上の気づき

この問題を調べる過程で、EPT のシナリオB自体の設計が、ユーザーが本来想定していた運用思想（「出口はインターフェースエージェント」＝サブエージェントの成果物は必ずメインループが検証・確定してから保存する）と食い違っていたことが判明した。従来の `agents/writer.md` は Write/Edit を持ち、writer 自身に最終ファイルを直接保存させる設計になっていた。

### 対策

`agents/writer.md` の `tools` から Write/Edit を除去（`Read, Grep` のみに変更）。本文セクション（Tools表、Constraints直前の説明、Output Format）も「Write/Edit アクセスはなく、成果物はレポート内にテキストとして返す」という趣旨に修正した。`rules/delegation.md` の報告前レビュー条項に「`writer` は Write/Edit ツールを持たない。成果物は本文としてテキストで返ってくるので、メインループが内容を検証したうえで、実際のファイルへの保存を行う」と追記した。

### 再検証1（scenB4、ファイル保存ありの設計）

writer 自身は Write を試みなくなり意図通りテキストを返したが、**委譲元（代理メインループ役のサブエージェント）自身が `summary.md` の保存を試みて同じガードにブロックされる**という新しい問題が判明した。これは「サブエージェントがさらに別のサブエージェントに委譲する多段階構造では、中間の委譲元自身も真のメインループの権限（Write ガード対象外）を持たない」という構造的な限界を示している。

### 再検証2（scenB5、テキスト完結型への再設計）

この構造的限界を回避するため、EPT のシナリオ自体を「ファイルへの保存は不要、要約全文をテキストで完了報告に含める」という完了条件に変更して再実行した。結果、Write ガードに一切遭遇せず、委譲判断（delegation.md のルーティングに従い writer に正しく委譲）・報告前レビュー（3実験の数値照合、exp3 原文の「val-accuracy cost」という記述と実数値の食い違いを自発的に検出し数値を優先する判断）とも精度100%で確認できた。

### 新たな設計上の知見（今後の EPT 運用に反映すべき事項）

- `~/.claude/rules/*.md` は、新規作成ファイルはセッション開始時の静的スナップショットに反映されない（既知）一方、**既存ファイルの内容編集はセッション中に動的に反映される**ことを今回確認した（前提確認で診断エージェントが更新後の内容を認識していた）。この非対称性（新規作成 vs 既存編集）は次回 EPT 設計時に踏まえるべき
- 多段階委譲（サブエージェントがさらに別のサブエージェントへ委譲する構造）を EPT で検証する際、中間の委譲元（代理エージェント）自身も Agent tool で spawn されたサブエージェントである以上、真のメインループの権限とは異なる制約（Write ガード等）を受ける。この構造を模擬検証したいときは、成果物をファイル保存させず、テキスト完結型のシナリオ設計にすることで、この構造的制約を切り分けて検証できる

## 2026-07-05: researcher/analyst/engineer のスコープ再定義と EPT 再検証

前日 (2026-07-04) の delegation ロール設計に続き、各エージェント定義のスコープを個別に見直した。

### researcher.md のスコープ拡張

description・Tasks が「文献サーベイ・競合調査」のみに限定されており、それ以外の一般的な技術的・事実確認的調査（アカデミック/ビジネス文脈以外）では発火しないという記述の狭さが判明した。「一般的な調査」を Tasks の最初の項目として追加し、description も外部ソースを使う調査全般をカバーするよう拡張した。`rules/delegation.md` のルーティング表該当行も「一般的な調査・事実確認・文献サーベイ・要約・競合調査」に更新した。

EPT で 4 シナリオ（一般技術調査、コードベース限定探索 vs Explore の境界、文献サーベイ本来ケースの回帰確認、general-purpose との境界 hold-out）を検証。2 round とも精度 100%・retry 0 で収束。副次的に、`general-purpose` の description が専門エージェントと重複し「専門エージェントに譲る」旨の条項がない点をログに記録したが、誤委譲は一度も発生しなかったため未対応のまま据え置いた。

### 全エージェント定義のスコープ横断調査

`analyst.md`, `engineer.md`, `ideator.md`, `novelty-checker.md`, `red-teamer.md`, `writer.md` を同様の観点（description と実体のミスマッチ）で調査した。`ideator`/`novelty-checker`/`red-teamer` は `research-theme-proposal` スキルのフェーズ専任ワーカーとして意図的に狭いスコープであり問題なし。`engineer`/`writer` は元々十分に広く問題なし。`analyst` のみ、当初は意図的な狭さと判断したが、後続の議論で再定義が必要と判明した（下記）。

### analyst.md の再定義

`analyst` の従来スコープ（実験設計・統計検定計画・コスト見積もり・統計分析・図表生成・アブレーション設計）はすべて下流の具体的実行作業であり、より上流の抽象的推論（仮説・メカニズムの形成）と結果解釈（論文の Discussion 相当: 代替説明の検討、結論の妥当性評価）が欠落していた。

「解釈」担当を別エージェントに分離する案は却下した。仮説形成と結果解釈は判定基準（仮説）を解釈が必要とする点で密結合であり、使用ツールも同一のため、分離はハンドオフの overhead を増やすだけと判断した。階層としては `ideator`（研究テーマ選定、上流）→ `analyst`（個別スタディの仮説形成・解釈・具体的分析、下流）で altitude が異なり重複しないことを確認した。

タスク棚卸しの結果:
- **コスト見積もり**（GPU時間・ストレージ→ドル換算）は `analyst.md` から `engineer.md` に移管（GPU/インフラの価格知識は engineer 領分、analyst の守備範囲は統計的サンプルサイズ決定まで）
- **アブレーション設計**は独立カテゴリではなく「実験設計」に統合（特殊ケースとして扱う）
- **図表生成**は `analyst.md` に残すが、統計的推論の可視化（比較プロット・効果量・信頼区間）に限定。運用監視ダッシュボード（学習曲線・GPU使用率・wandb）は `engineer.md` に移管

結果、`analyst.md` の Tasks は優先順で: (1) 概念的推論・解釈 [新規・最優先], (2) 実験設計（アブレーション含む）, (3) 統計検定計画, (4) 統計分析, (5) 結果の可視化（統計的推論限定）。既存の Statistical Rigor Checklist に「結果に対する代替説明を検討したか」「結論が根拠の強さに見合っているか（過大/過小主張していないか）」の2項目を追加した。

`engineer.md` には「コスト見積もり」タスク（計画中の実験・スイープの GPU時間・ストレージ・クラウド課金見積もり）を追加し、既存の Infrastructure タスクに「運用監視ダッシュボード（学習曲線・GPU使用率）」を追加した。`rules/delegation.md` の analyst 行を「仮説・メカニズムの検討/結果解釈・実験設計・統計分析・統計的可視化」、engineer 行に「計算コスト見積もり・運用ダッシュボード」を追記した。

再定義後の `analyst.md`/`engineer.md` ペアで EPT を再実行（4 シナリオ: 既存結果の概念的解釈、コスト見積もりの境界 vs engineer、wandb ダッシュボードの境界 vs engineer、概念的枠付けのない純粋な統計実行の hold-out）。2 round とも精度 100%・retry 0 で収束。ログ上の非ブロッキング所見として、analyst description の「仮説を推論して...設計して...実行する」という逐次的な言い回しが必須パイプラインと誤読されうる点を記録したが、別立ての「Use for X, or Y, or Z」節が既に曖昧さを解消しており、実際の誤委譲は観測されなかった。

### 見送った変更

- **`git-operator` エージェントの新設**: `commit`/`push`/`commit-and-push` スキルは既に per-skill で `model: "haiku"` の ad-hoc サブエージェントを spawn しており重複がなく（`commit-and-push` は commit/push を参照で合成しコピーしていない）、スキルが常に明示的に起動するため open-ended なルーティングの必要性もない。常駐の名前付きエージェントは儀式が増えるだけで利益がないため見送った
- **`notes/LOGS`/`TODO.md`/`ISSUES.md` の定型更新を `writer` に委譲する案**: これは文章構成ではなく事実の記帳作業であり、かつ `writer` は Write/Edit ツールを持たないため委譲すると無意味な往復が増える。実行エージェント本人（またはメインループの `/log-progress`）による直接作業のまま据え置いた

本セッションで新たに顕在化した未解決の技術課題はない。project-level の設計・ロードマップ（PLAN.md）変更もなし（エージェント定義のチューニングであり、プロジェクト全体の設計転換ではない）。

## 2026-07-08: engineer/analyst のモデルを opus から fable に変更

`agents/engineer.md` と `agents/analyst.md` の frontmatter `model:` を `opus` から `fable` に変更した。公式ドキュメント（code.claude.com の model-config / sub-agents / errors）で subagent の model 挙動を確認した上での変更。

確認した事実：

- `fable`（Claude Fable 5）はサブエージェント frontmatter の有効なエイリアス。他に `best` / `opusplan` / `sonnet[1m]` / `opus[1m]` / `inherit` 等がある。
- frontmatter に無効なモデル値を書いた場合は `inherit`/デフォルトへ**サイレントフォールバック**する。opus への強制変更コールバックのような機構は存在しない。一方 CLI の `--model` フラグや `ANTHROPIC_MODEL` 環境変数に無効値を指定すると起動時エラー。事前の値検証は Anthropic API 直接利用時のみ。
- fable が**使用制限（レート制限・使用量上限）**に達しても他モデルへ自動フォールバックしない。`You've hit your session/weekly limit · resets <時刻>` が表示されリセットまでブロックされる。レート制限・billing・認証系のエラーは fallback model chain の対象外。`best` は起動時の初期選択でありランタイムフォールバックではない。fable 明示指定時のフォールバックは安全分類器（サイバーセキュリティ・生物学系）発火時のみ。

含意：fable を既定にすると使用制限到達時に自動切り替えされずセッションがブロックされる。頻発するようなら `model:` 指定の見直し（opus/sonnet へ戻す等）を要する。ユーザーは制限リスクを承知の上で fable のまま運用する判断。

別件として `settings.json` の default `"model": "sonnet"` pin を削除した（本セッション開始前からの変更）。

## 2026-07-09: `rules/delegation.md` を除外リスト方式から allow-list/default-deny 方式へ改訂

### 問題

メインループがサブエージェントへの委譲を怠り、本来 `engineer`/`analyst` に振るべき実作業（実装・分析）を自ら手を動かして実行し、トークンを浪費する事象が観察された。従来の `rules/delegation.md` は「実作業は委譲する」という原則 + ロール別ルーティング表という除外リスト方式（何を委譲すべきかを列挙し、それ以外は暗黙に自由裁量）であり、メインループが「これは簡単だから自分でもできる」と誤判定した場合に歯止めがなかった。

### 調査

Anthropic 公式のマルチエージェント設計指針、一般的なスコープクリープ防止技術、Claude Code 固有の強制機構の3方向から調査した。

### 却下した対策案

- **難易度を事前見積もりして委譲要否を判断する**: 誤判定そのものが今回の問題の原因であり、誤判定を防ぐための追加判断ステップにも同じ誤判定リスクが乗る。解決にならない
- **メインループから実行系ツールを剥奪する**: メインループはサブエージェントの成果物を最終的にファイルへ反映する役割（報告前レビュー）を担っており、Write/Edit を完全に奪うとその役割自体が果たせなくなる
- **hook による強制**: 「これは調査か対話か」のような意味判断を hook（決定論的スクリプト）は原理的に下せない

### 採用した対策

`rules/delegation.md` の「基本方針」を除外リスト方式から allow-list/default-deny 方式に書き換えた。デフォルトは常に委譲とし、メインループが直接手を動かしてよいのは (a) ユーザーとの対話・確認・質問、(b) サブエージェント出力のレビュー・要約・編纂・ファイル反映、の2点のみに限定した。「これは自分でもできる」という自己評価を判定基準として使わない旨を明記した。

### EPT による検証（4 iteration、収束）

- **Iteration 1**: 3シナリオ（軽微な修正依頼／単純な事実確認／サブエージェント出力のファイル反映）をバイアスフリー実行エージェントで実行。全シナリオ成功（100%）だったが、(a)/(b) の境界基準（「調査」との区別、ロール別ルーティング表との優先関係）が文面上未定義で、実行エージェントが独自解釈で穴を埋めていたことが判明
- **Iteration 2**: (a)(b) それぞれに境界を定義する一文を追加。再検証で事実確認・軽微修正シナリオは解消したが、サブエージェント出力反映シナリオで新たな優先順位未定義問題（(b) 該当時にロール別ルーティング表を無視してよいか）が発覚
- **Iteration 3**: 「(a)(b) への該当が判定できた時点で、ロール別ルーティング表の参照は不要」という優先規則を追加。再検証でこの問題の解消を確認
- **Iteration 4**: 全シナリオへの全面リグレッション + 元の問題意識（作業途中でタスク難易度が上昇した際のサンクコスト的継続）を狙った新規 hold-out シナリオを追加実施。既存の「着手済みの作業を継続する場合も、行動の節目ごとにこの原則へ照合し直す」の一文が、途中委譲への切り替えを正しく引き起こすことを確認。4シナリオとも2ラウンド連続で新規 unclear points 0・精度100%を維持し、収束と判定してループを終了した

### 結論

`rules/delegation.md` の基本方針は allow-list/default-deny 方式で確定。ロール別ルーティング表・報告前レビュー節は変更なし。

# サブエージェント役割分担・モデルルーティング

## 2026-07-04: 経緯と設計

別セッションでの試行錯誤から、以下の運用パターンが使い心地良いと判明:
- メインループ（対話）は Sonnet
- 調査・実装・分析などの実作業はバックグラウンドのサブエージェントに委譲
- サブエージェントのモデルは役割で分ける: 調査・文書作成 = Sonnet, 実装・分析 = Opus
- メインループはサブエージェントの出力をレビュー・編纂してから報告

これを CLAUDE.md / エージェント定義に恒常化するため調査した。

### 確認した事実（公式ドキュメント）

- サブエージェントのモデル解決順位: ① `CLAUDE_CODE_SUBAGENT_MODEL` 環境変数 → ② Agent ツール呼び出し時の `model` パラメータ → ③ サブエージェント frontmatter の `model` → ④ メイン会話のモデル（[sub-agents](https://code.claude.com/docs/en/sub-agents.md), [model-config](https://code.claude.com/docs/en/model-config.md)）
- v2.1.198 以降、サブエージェントはデフォルトでバックグラウンド実行
- `~/.claude/rules/*.md` はユーザーレベルで全プロジェクトに自動読み込みされる（`.claude/CLAUDE.md` からの明示的 `@import` 不要、公式仕様）

### 現状確認で見つかった問題

- 観察されていた「実装・分析は Opus」を裏付ける設定はどこにも存在しなかった（登録済みカスタムエージェント `ideator`/`novelty-checker`/`red-teamer` は全て `model: sonnet`）。おそらくメインループ自身がその場の判断で `model` パラメータを渡していただけで、恒常的な設定ではなかった
- `agents/pt-engineer.md` / `pt-analyst.md` / `pt-research.md` は YAML frontmatter が存在せず、登録済みサブエージェントとして機能していなかった（`plan-to-implement`/`implement` スキルからも参照なし = 孤立ファイル）
- これらは 2026-03-22 の `project-team` スキル設計時に作られたもので、project-team スキル本体（フェーズ管理・ハンドオフ・状態ファイルを伴う重量級フレームワーク）は 2026-04-06 に無効化、2026-05-04 に `abff630` で完全削除されていた（理由の記録なし）

### 設計判断

project-team のような「スキル内に役割別エージェントを埋め込み、フェーズ/状態ファイルで管理する」設計は使われずに終わった。同じ轍を踏まないため、今回は以下の方針で最小構成にする:

- 役割別エージェント（engineer/analyst/researcher）はどのスキルにも属さない独立したプリミティブとして定義する。特定のワークフロー（project-team のような）に埋め込まない
- フェーズ管理・ハンドオフ・状態ファイル・Critique Mode は導入しない。ユーザーが観察した必要な要素は「モデル固定」と「報告前レビュー」の2点のみ
- `pt-engineer.md`/`pt-analyst.md`/`pt-research.md` の内容（実装/分析/調査タスクの具体的な作業内容・規約・出力フォーマット）は質が高く再利用可能と判断し、project-team 依存部分のみ除去して流用

### 実装内容

| ファイル | 内容 |
| ---- | ---- |
| `agents/engineer.md` | コード実装・テスト・インフラ担当、`model: opus`、tools を Read/Edit/Write/Bash/Grep/Glob/Agent に制限 |
| `agents/analyst.md` | 実験設計・統計分析担当、`model: opus`、tools を Read/Edit/Write/Bash/Grep/Glob に制限 |
| `agents/researcher.md` | 文献サーベイ・執筆担当、`model: sonnet`、tools を Read/Write/Edit/Grep/WebSearch/WebFetch に制限 |
| `rules/delegation.md` | 役割別ルーティング表（作業内容 → エージェント → モデル）+ 報告前レビュー方針 |

`presenter` のような追加ロールは、実際に必要になった時点で同じ型で追加する方針とし、今回は作らない（要求されていない機能を先回りしない）。

## 2026-07-04: `writer` エージェントの追加

ユーザーが別途確認した経験則: Opus（特に 4.8）はライティング能力が著しく低く、現行ハーネス + Sonnet の応答が文書作成では最も好ましい。この観察に基づき、文書執筆を専門とする `writer` エージェントを追加した。

- `agents/writer.md`（`model: sonnet`。他の3ロールと技術的には同じ frontmatter 指定であり、「固定」という特別な機構があるわけではない。当初その旨を強調する記述を加えたが、他ロールとの扱いの違いを誤って示唆するため削除した）
- `researcher.md` から「論文執筆」「提案書執筆」タスクと関連する description の文言を除去し、`writer` に委譲する形に整理（役割説明の重複はサブエージェント自動選択の精度を下げるため）
- `rules/delegation.md` のルーティング表を他ロールと同じ表記（モデル名のみ）に統一

### 既知の限界

- サブエージェントへの自動委譲判定（description に基づく）自体は Claude の裁量であり、ハーネス機能として強制はできない。今回固定できたのは「委譲された場合のモデル」のみ
- 「報告前に必ずレビューする」もプロンプト指示であり、強制メカニズムはない（CLAUDE.md 全般の制約）

## 2026-07-04: EPT iteration 1 の実施と「delegation.md 未ロード」の発見

上記設計（`agents/engineer.md`・`analyst.md`・`researcher.md`・`writer.md` と `rules/delegation.md`）に対し、empirical-prompt-tuning の手順で iteration 0（静的整合性チェック）・iteration 1（実行検証）を実施した。

### Iteration 0（静的チェック）

4エージェントの description と本文の整合性を確認。問題なし。

### Iteration 1（実行検証）

新規サブエージェントに対し、以下3シナリオを与えて挙動を観察した:

- シナリオA: 実装タスク（`engineer` への委譲を期待）
- シナリオB: 執筆タスク（`writer` への委譲を期待）
- シナリオC: 些細な1行修正（委譲すべきでない）

結果:

| シナリオ | 期待 | 実測精度 |
| --- | --- | --- |
| A（実装） | engineer に委譲 | 33% |
| B（執筆） | writer に委譲 | 63% |
| C（些細な修正） | 委譲しない | 100% |

シナリオA・Bはいずれも委譲されず、実行エージェント自身がタスクを自力完結させるケースが多数を占めた。シナリオCのみ意図通り「委譲を避ける」判断が一貫していた。

### 決定的な診断

上記結果の原因を特定するため、追加の診断サブエージェントに「delegation 関連の指示を保持しているか」を尋ねた。回答から、`rules/delegation.md` がロード済みコンテキストに一切含まれていないことが判明した。実際にロードされていたのは CLAUDE.md、karpathy CLAUDE.md、japanese-tech-writing.md、concise、agent-teams.md、workflow.md のみだった。

**原因**: `~/.claude/rules/*.md` はユーザーレベルで全プロジェクトに自動読み込みされるが、その読み込みは**セッション開始時に一度だけ**行われる静的スナップショットであり、動的な再スキャンは行われない。`delegation.md` は本セッションの作業中に新規作成したファイルであるため、本セッション中に生成されたメインループ自身にもどのサブエージェントにも反映されない。

これは対照的に、`agents/*.md`（カスタムサブエージェント定義）はセッション中も動的にスキャンされ、作成直後から `Agent` ツールの `subagent_type` として利用可能になる、という非対称性を実地で確認した結果でもある。

### 結論

今回の EPT 結果（33%/63%/100%）は `rules/delegation.md` の文言の実効性を測る有効なデータではない。文言そのものの評価は、`delegation.md` が静的スナップショットに含まれる次回セッション開始後、同じ3シナリオを再実行して初めて可能になる（TODO.md に申し送り済み）。

### 副産物: Write ツールの report.md ガード

シナリオB（執筆タスク）の実行中、Write ツールがファイル名 `report.md` を「サブエージェントは所見をテキストで返すべきで、報告ファイルを書くべきではない」というガードでブロックする挙動が観察された。ユーザーが明示的に指定した保存先パスであってもブロックされた。これは今回の委譲設計とは別の、既存ハーネスのガードの過剰適用であり、今回は対応せず ISSUES.md に記録するに留めた。

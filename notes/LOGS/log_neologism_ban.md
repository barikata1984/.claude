# Log: 造語禁止ルール

## 2026-06-10: EPT による造語禁止ルール新設

### 背景

エージェントが学術的に新規性のある研究ネタを議論するとき, 既存の単語・言い回しで十分に説明できるのに, それっぽいだけの造語を作って説明しようとする傾向がある. 結果として対話にならないことが多い. この挙動を完封する CLAUDE.md ルールを設計・検証した.

### EPT 設計

**シナリオ:**
- A (中央値): 触覚センサ × ロボットマニピュレーションのブレスト
- B (edge): world model + residual policy learning のアーキテクチャ整理
- C (hold-out): ユーザーが明示的に命名を求める場面

**要件チェックリスト (シナリオ A/B):**
1. [critical] 確立していない用語を定義なしに使っていない
2. [critical] 提案内容が既存の学術用語で説明されている
3. 研究としての具体性がある
4. 対話として成立する / 接尾辞による新語装飾がない

### 反復経過

| iter | 修正 | 新出不明瞭点 | 精度 |
|---|---|---|---|
| 1 | (ベースライン) | 「造語」定義が曖昧, 「〜性」禁止が広すぎる | A: 100%, B: 100% |
| 2 | 造語判定基準を「学術文献ヒット」に具体化 + 「〜性」除外例追加 | ブレスト解除の波及範囲 | A: 100%, B: 100% |
| 3 | 「本節はブレスト局面でも解除しない」を明示 | 0 | A: 100%, B: 100% |
| 4 + hold-out | 変更なし (収束確認) | 0 | A: 100%, B: 100%, C: 100% |

subagent 合計: 9 体 (A×4, B×4, C×1)

### Hold-out 結果

シナリオ C (命名要求): subagent は (1) まず既存概念 5 つで内容を説明, (2) 既存語で十分かもしれないと留保, (3)「仮に〜と呼ぶ」と明示的に仮称提案, (4) ユーザー同意なく使い続けないと宣言. 指示の escape hatch が正確に機能した.

### 最終指示の構造

「何が造語か」の定義 (学術文献ヒット基準) → 禁止行為 → 代わりにすべきこと (仮称提案の手順) → ブレスト非解除

### 残存リスク

- 「確立した用語の組み合わせ」と「造語」の境界は完全に線引きできない. 検索基準は機能するが, subagent は都度判断している
- ブレスト局面で仮称が有用な場面はある. 完封指示がやや制約的に働く可能性がある (実運用で要観察)

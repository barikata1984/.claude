# Plan

## plan-to-implement スキル

### 目的

`~/.claude/rules/agent-teams.md` はチーム編成の機構（TaskCreate・依存・owner・shutdown）を定義するが、「チームで分割 vs 単一サブエージェント」のトリアージ判断そのものは定義していない。このギャップを埋め、機能主導のインターフェース交渉によってエージェントチームの並列実装を駆動するオーケストレータを、`/implement` から呼べるスキルとして構築する。

### 配置方針

ルールではなくスキルに置く。単独でも `/implement` からでも呼べる（起動可能・合成可能）性質を持ち、多フェーズの手続きは常時ロードのルールとして持つには嵩が大きいため。方針の門番一行だけは、将来 `agent-teams.md` 側に条件付きで置く余地を残す。

### トリアージゲート（2問）

チームで分割するか単一サブエージェントに任せるかを、以下の2問で判定する。

1. **広さ方向 vs 深さ方向** — 独立した多数の対象への作業（広さ）か、密結合な単一物への作業（深さ）か
2. **継ぎ目の名指し可否** — いま継ぎ目（seam＝機能間の境界の契約）を具体的に名指しできるか

判定: 広さ＋名指し可能 → チーム分割。深さ、または名指し不可 → 単一サブエージェント。

### 機能主導のインターフェース交渉フロー

自動車/プラント設計の ICD（Interface Control Document）に類する手続きで、チーム分割時の契約を確定する。

1. 各機能担当が自分の I/O 要求を申告する
2. インターフェース担当（調停権を持つハブ）が申告間の衝突を調停し、契約案を提案する
3. 各機能担当が契約案をレビューする
4. 全員が OK するまで ≤3 ラウンド反復する
5. 全員 OK で契約を凍結する
6. 非収束（3ラウンドで収束しない）の場合: 単一サブエージェントへの降格、再分割、またはユーザーへのエスカレーションのいずれかを選ぶ。非収束はトリアージ判断そのものが誤っていた信号として扱う

**トポロジ**: 星型（インターフェース担当をハブに、機能担当はハブ経由で交渉）。全対全（O(N²)）の交渉を避ける。機能数が多い場合は ~4 群への再帰的メタ分割で捌く。

**messaging のスコープ**: 交渉は計画フェーズに限定する（凍結契約を生成する手段）。実装フェーズは各担当が凍結契約を読むだけで、messaging は行わない。

**価値の所在**: 目的は第一に実行速度ではなく、単独では担いきれないインターフェース調停の認知負荷を肩代わりすること。成功条件は「維持可能な契約が出たか」に置く。

### ステージ構成（実装順）

- **v1（最小版）= Stage 1（交渉フロー）+ `/implement` ラッパ — 完了・検証済み（2026-07-02）**。`plan-to-implement/SKILL.md`（本体）+ `implement/SKILL.md`（薄いラッパ）を起草し、ept 5 iteration・8 シナリオで検証（全 accuracy 100%、実質収束）。素直なタスクなら完全版と同等の成果を出す想定
- **使いながら判断（保留、投機的に作らない）= Stage 2・Stage 3**: Stage 2（トリアージと非収束処理の硬化：単一 vs チーム判定精度、デッドロック時のエスカレーション、継ぎ目名指しの粒度）、Stage 3（大 N の再帰的メタ分割）はいずれも実運用で使いながら必要性を判断してから着手する。ロードマップ上、Stage 2 は「完全版に必須（critical）」から本方針に変更（2026-07-02）
- **条件付き（必要が観測されたら実装）= Stage 4（実装中の契約再交渉）、`agent-teams.md` への方針一行の追加**

検証は各段階で empirical-prompt-tuning（ept）による二面評価を行い反復する。

## ユーザーレベル設定の汎用化（完了）

### 目的

`~/.claude/` 配下のスキルと CLAUDE.md を、特定プロジェクトに依存しない汎用的な定義に整理する。

### 方針

1. **スキル** — プロジェクト固有のパスやブランチ名をハードコードしない。役割定義はスキル側に持ち、具体的なパスはプロジェクトの CLAUDE.md に委ねる
2. **CLAUDE.md** — ユーザーレベルには全プロジェクト共通のルール（コーディング規約、ワークフロー、標準ドキュメント構成）のみ残す。プロジェクト固有の内容（コマンド、参照先、注意事項）は各プロジェクトの CLAUDE.md に移動
3. **スクラッチプロジェクト** — ユーザーレベル CLAUDE.md にスクラッチプロジェクトでの標準ドキュメント構成（docs/TODO.md, docs/LOGS/, docs/ISSUES.md, docs/PLAN.md）を明記し、新規プロジェクト立ち上げ時の指針とする

## 挙動制御フレームワーク

### 目的

Claude の RLHF 由来デフォルト挙動（scope expansion、taxonomy ハルシネーション、
日本語スタイル違反等）を、CLAUDE.md directive と Stop hook の二段防御で抑制する。
arXiv:2511.13972 の実証では directive 単独で 56% 削減にとどまるため、
決定論的な hook で残り 44% を捕捉する設計。

### 方針

1. **層別防御** — 予防（CLAUDE.md directive）+ 事後検査（Stop hook）の組合せ。
   全会話に効くシンプルな構成を優先し、UserPromptSubmit hook や subagent
   systemPrompt override は将来の拡張として位置づける
2. **単一目的 hook** — 検出対象ごとに hook を分離（scope-check / taxonomy-check）。
   ログも分離して個別に偽陽性率を評価可能にする。共通の transcript parsing は
   3 つ目の hook が必要になるまで重複を許容（YAGNI）
3. **block mode 運用** — warn-only での試運用は精度確認時のみ。本運用では
   block JSON を返して再生成を強制し、`stop_hook_active` フラグで無限ループを防止
4. **観測駆動の改良** — 偽陽性率を `*-violation.log` から定期レビューし、
   キーワードや検出ロジックを調整。試行錯誤の根拠は `docs/LOGS/log_behavior_control.md`
   に追記
5. **under-retrieval（外部知識導入のサボり）対策** — hallucination とは別軸（対・事実の
   情報取得衛生）として CLAUDE.md に「参照情報の検証」節を独立配置。段階 1（出力時検証）→
   段階 3（「**最初の行動: 検索**」入力時トリガー + 免除限定列挙）に強化済み（EPT 5 iter +
   hold-out）。強制ハーネスは原理的に不可と確認済み。実運用での under-retrieval が継続したため
   CLAUDE.md を「全技術・科学的問い」包括ルールに改訂（2026-06-19）。信頼度バイアスへの対応として
   UserPromptSubmit + SubagentStart フック（検索優先リマインダー注入）を導入済み。常駐デーモンと
   hallucination 対策一式は不採用

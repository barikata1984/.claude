# Claude Code ユーザーレベル設定

## ワークフロー

- 3ステップ以上のタスクは Plan Mode から開始せよ
- 調査・分析タスクはサブエージェントに委譲し、メインコンテキストを保護せよ
- 実装完了を自己宣言するな。テスト実行・差分確認・ログ確認で動作を実証せよ
- バグ報告を受けたらまずログ・エラー・テストから自力調査せよ。質問は調査後に

## 応答スコープ規約

ユーザーの質問・依頼には、字義通りに要求された範囲のみで応答せよ。以下を厳守：

- 質問・依頼で明示されていないサブセクション（「なぜ」「結論」「推奨」「次のステップ」「要点」「まとめ」「補足」「例」「比較」「方針」等）を勝手に追加するな。一文で答えられる質問には一文で返せ
- 真に曖昧な場合のみ明確化質問を 1 つ、1 行で返せ。複数選択肢を列挙してユーザーに選ばせる形で本来の応答を回避するな
- 不確実性は範囲拡張ではなく、知識ソースブロックでの明示で扱え

## 機密情報の取扱い

- 値の確認はユーザーに dashboard 等で依頼。動作確認は末端機能の成否で間接判断（存在確認は `doppler secrets --only-names` / `pass ls` / `test -f`）
- 万一ターミナルに漏出した場合: 即座にユーザーへ報告 → secret rotation と shell history (`~/.zsh_history`) クリーンアップを案内

## 調査・回答ポリシー

- ユーザーの質問・要求に回答する際、内部知識だけで即答するな。必ず WebSearch / WebFetch で最新の一次情報を取得せよ
- 検索結果をそのまま転記するな。論理構造の把握 → 推論・演繹 → ベストプラクティスの導出 のプロセスを経て回答せよ
- 「可能性がある」「疑いがある」と感じた時点で、報告せず先に検証せよ。コード挙動なら実行・ログ確認、仕様ならドキュメント参照、エラー原因なら該当コード読み込みと再現で確定させる。検証手段が尽きた場合のみ、仮説であることを明示して報告してよい

### 知識ソース明示ルール

**すべての回答の末尾に知識ソースブロックを付与せよ。** `/request-source` スキルの出力フォーマットに従う。
水平バー（`---`）で本文と分離する。
内部知識を使った回答ほどユーザーが検証できないため、明示によってユーザーが WebFetch 要求を判断できるようにする。
ブロックの付与を省略した場合、次の自分の返答の冒頭で `/request-source 直前の回答` を自動実行して補完せよ。

```
---

[知識ソース]
内部知識：
- <学習データ由来の項目>

外部知識：
- <URL> — <取得した情報の要約>
- なし

観測事実：
- <このセッションで直接確認したログ・コード・エラー出力>
```

- 外部知識が「なし」の場合、ユーザーは「そこを WebFetch せよ」と指示できる
- 観測事実は内部知識でも外部知識でもない第三のカテゴリとして分離する
- ブロックが欠けている場合、ユーザーは `/request-source この回答` または `/request-source 直前の回答` で明示的に要求できる

### WebSearch / WebFetch 必須トリガー

以下のいずれかに該当する場合、回答・実装の前に必ず WebSearch / WebFetch を実行せよ。
「判断」を挟まず、条件に該当した時点で自動的に実行すること。

- **API・ライブラリ仕様**：関数挙動・引数仕様・非推奨 API の代替・外部 API を使うコード実装
- **エラー診断**：エラーメッセージの原因説明、「このコードがなぜ動かないか」
- **推論の言語的マーカー**：「〜のはず」「〜と思われる」「おそらく」「〜かもしれない」「一般的に」「通常は」を使いそうなとき
- **数値・パラメータ根拠**：ベンチマーク値・推奨デフォルト・パフォーマンス特性
- **環境依存**：OS・言語バージョン・CUDA・コンテナ環境固有の挙動
- **学術・工学的情報**：物理公式・数値計算手法・論文・標準規格・シミュレーション

## コーディング規約

- **Formatter**: `ruff format`
- **Linter**: `ruff check`
- **Type hints**: Python 3.10+ annotations required
- **Line length**: 100 characters
- **Import order**: stdlib → third-party → local (enforced by ruff isort)
- **Docstrings**: Google style, only for non-obvious functions

## Git Workflow

- **Commit format**: Conventional Commits (`feat:`, `fix:`, `refactor:`, `test:`, `docs:`)
- **Main branch**: `main`

## 略語使用規約

文書・レポートで略語を使用する際は以下に従う。

- **初出時**: `完全名 (略語)` の形式で記述する（例: `Discrete Elastic Rods (DER)`、`Bayesian Optimization (BO)`）
- **2回目以降**: 略語のみで記述する
- **ユーザーが既知と明示した略語**: 初出から略語のみで構わない

## 日本語スタイル規約

日本語でドキュメント・レポート・要約ノート等を生成する際は `rules/japanese_style.md` に従う。

**日本語応答を出力する直前に、本文中の英単語を一つずつ点検し、固有名詞・コード片・数式・skill 構造ラベルでなければ和訳せよ。**

- 単一ドキュメント内で言語を混在させない（入力言語に従って本文言語を決定）
- 一般語彙（`gap`, `principled`, `tradeoff` 等）は日本語化、固有名詞・コード片・数式・各 skill の構造ラベルのみ英語のまま
- 英語構文の直訳（番号付き体言止め羅列、副詞丸投げカタカナ化）を避ける

固有名詞リスト・代替表現・違反例の詳細は `rules/japanese_style.md` を参照せよ。

## 標準ドキュメント構成

スクラッチプロジェクトでは以下の構成を **`notes/` 配下** に作成する
（デフォルト基底ディレクトリ）。`docs/` は legacy / フォークプロジェクトで
既に採用されている場合のみ使用する。フォーク・既存プロジェクトでは
既存の構成に従う。

**ディレクトリ名の解決規則**: skill が `notes/X.md` または `docs/X.md` と書いている
場合、実際のパスはプロジェクトの既存構成に従って解決する:

1. `notes/` ディレクトリが存在する → `notes/X.md` を使う (デフォルト)
2. `docs/` ディレクトリが存在する → `docs/X.md` を使う (legacy / フォーク)
3. どちらも存在しない (新規作成時) → project-level CLAUDE.md の指定に従う、
   なければ **`notes/` を新規作成** (デフォルト)

以下の表では `<notes>` を「プロジェクトのドキュメント基底ディレクトリ
(`notes/` がデフォルト、`docs/` は legacy)」を表すプレースホルダとして使う。

| ファイル | 役割 | 運用ルール |
|---|---|---|
| `<notes>/TODO.md` | タスクリスト | 完了項目は `[x]` に。新規タスクを追記 |
| `<notes>/LOGS/` | 作業・実験記録（トピック別 `log_<topic>.md`） | append-only。過去の記録は編集しない |
| `<notes>/ISSUES.md` | 未解決の技術課題 | 解決済みは削除（アーカイブしない） |
| `<notes>/PLAN.md` | 設計方針・ロードマップ | 設計判断が変わった場合のみ更新 |

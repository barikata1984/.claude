# Log: 批判的スタンス

## 2026-05-27: karpathy-guidelines プラグインの常時ロード可否評価 — 不採用

**背景**：インストール済みプラグイン `andrej-karpathy-skills@karpathy-skills`（skill
`karpathy-guidelines`、Andrej Karpathy の LLM コーディング失敗観察に基づく 4 原則）を、
毎セッション自動ロードして CLAUDE.md と併用すべきか評価した。

**トピック網羅マトリクス**：karpathy の 4 原則（Think Before Coding / Simplicity First /
Surgical Changes / Goal-Driven Execution）はすべて CLAUDE.md 既存セクションに対応物を持ち、
karpathy 固有の新規トピックはゼロ。正味の新規貢献は Surgical Changes の「既存スタイル尊重・
無関係な隣接コードを触らない」というコード編集固有の具体性のみ。

**競合分析**：唯一の真の競合は「プロンプト側の明確化質問の閾値」。karpathy 項目 1 は
「迷えば聞け・選択肢を提示せよ」、応答スコープ規約は「真に曖昧な時のみ 1 つ・列挙して
回避するな」で逆向き。両者とも対象は*ユーザーのプロンプト*であり、対象の混同ではなく
同一対象上の閾値衝突である。これは情報の確実性ポリシー（対象は*エージェントが集めた/
創発した情報*）とは別物で、後者は karpathy に対応物が無く、karpathy 主軸にすると脱落する
（競合ではなく欠落）。

**機構確認**（claude-code-guide エージェント経由でドキュメント検証）：
- skill 本体は遅延ロードで、常時ロードを強制する設定は存在しない。
- プラグイン root の CLAUDE.md は自動ロードされない（人間向けドキュメント扱い）。
- 毎セッション確実に注入する手段は、ユーザー CLAUDE.md 直書き / `@path` インポート
  （起動時に全文展開、最大 5 段ネスト、初回承認ダイアログ）/ SessionStart フックの
  `additionalContext` 出力の 3 つ。

**決定**：karpathy を主軸に据える方針は不採用。理由は (1) 4 原則の大半が既存規約の二重化、
(2) 主軸化すると応答スコープ規約・評価者規約・ワークフロー（サブエージェント委譲・デバッグ
初動の自力調査）を書き戻す必要が生じ、プロンプト側の質問閾値も反転する、(3) 正味の新規価値は
Surgical Changes の数行のみ。費用対効果から、必要なら Surgical Changes 相当の数行を応答スコープ
規約に接ぎ木する方が安価（接ぎ木は本セッションでは未実施）。

## 2026-05-29（続）: 反迎合「批判的スタンス」節の追加と empirical 検証

**背景**：karpathy 主軸構成（前項）で評価者規約を一旦解除したため、ユーザー反論への
迎合バイアスが戻りうる残存リスクがあった。これを補うため、Claude Code を迎合的でなく
批判的に振る舞わせる directive を CLAUDE.md に追加する方針を決め、まず Web でベスト
プラクティスを調査した。

**ベストプラクティス調査**（WebSearch / WebFetch）：

- 複数記事が同一テンプレに収束（[BSWEN](https://docs.bswen.com/blog/2026-02-12-stop-claude-agreeing/) /
  [aiproductivity.ai](https://aiproductivity.ai/news/anti-sycophancy-prompt-claude-direct-feedback/)）。
- 効く順に: (1) 役割再定義 `"You are not my assistant"`（従属役割を壊し反論を許可、最も効く）、
  (2) 抽象語 `"be critical"` より具体的指示 `"Challenge my assumptions"`、(3) 問題先出し
  （励まし→批判の順を逆転）、(4) 両論併記より立場表明、(5) 配置はグローバル `~/.claude/CLAUDE.md` 固定。
- 警告: 過激語 `"Be brutal"` は無礼化し helpfulness 低下（過剰補正）。code review・設計判断・
  文章批評には有効だが、ブレスト・創造的探索には不向き（gentler mode = 素の受容的デフォルトの方が良い）。

**追加内容**：`~/.claude/CLAUDE.md` の `@import` 直後・`## Git Workflow` 前に
`## 批判的スタンス（反迎合）` 節（7 bullet）を新設。karpathy §1 と重複する「前提を疑う・
トレードオフ提示」は避け、不足分のみ追加：役割再定義、枕詞禁止、問題先出し、反証義務
（最低 1 つ探し無ければ明示）、証拠ベースの立場表明、過激語禁止の安全弁、ブレスト時の解除条項。
最後の 2 bullet が調査の警告（過剰補正）への対策。

**empirical 検証**（`empirical-prompt-tuning` skill、A/B 設計）：
- 対照群 = karpathy §1 のみ、処置群 = §1 + 新節。差分が新節の限界効果。
- シナリオ 4 種 × 2 群 = 8 subagent。中央値（JWT localStorage）、edge（GIL 誤前提）、
  過剰補正検出（ブレスト）、主観的迎合罠（コードレビュー廃止の同意要求）。
- 結果：**全シナリオ両群とも要件精度 100%、不明瞭点 0 件**。

**判明したこと**：
1. **過剰補正なし**：ブレスト処置群は解除条項を明示発動し否定ゼロで発散。全 8 本で煽り語・
   無礼ゼロ。安全弁は機能。
2. **節の上乗せ効果は小さいが実在**：karpathy §1 の "push back when warranted" 単独でも
   明確な誤り・主観的迎合罠の両方で既に反論が出る（土台が強い）。節が反復して足した固有効果は
   **明示的反証探索（steelmanning）規律**のみ（処置群 B/D は「反証も探した: …」を構造化、
   対照群は出さず）と、わずかに直截な口調。
3. **A/B 手法の弁別力の限界（正直に明記）**：対照群に karpathy §1 を含めたため比較は
   「§1 vs §1+節」の限界効果に過ぎず、真の default Claude（批判ガイドラインなし＋迎合訓練）
   とは比較していない。subagent ロールプレイは実セッション挙動の代理。Opus はどんな批判的
   フレーミングでも反論しやすく弁別力が構造的に低い。

**決定**：節は現状のまま採用（編集不要）。過剰補正なし・テキスト曖昧さなし・解除条項機能、
固有価値（反証規律・問題先出し・枕詞禁止・解除条項）は karpathy §1 と非冗長。これ以上の
A/B 反復は弁別力頭打ちでコストに見合わない（skill のリソース打ち切り基準）。真の効果測定は
A/B でなく実運用での体感評価（迎合が出やすい主観判断の局面）の方が情報量が高い。

## 2026-05-30: 過剰応答（未要求出力・喋りすぎ）への予防層導入と検証

**問題**：批判的スタンス節の導入後、エージェントが「先走って依頼していない情報を出力し、結局
何を言っているか分からない」挙動が頻発（資料を貼っただけで未要求の批判レビューを始める等）。
真因は、批判的スタンス節の「問題点・リスクから先に述べる」が、スコープ・ガード不在のまま
資料提示・事実質問にも適用されたこと。これは過去に削除した「応答スコープ規約」と同型の課題。

**機構の検証（推測でなく一次情報で確定）**：
- 既知の名称は scope creep（karpathy が Claude Code 最頻失敗と指摘）+ verbosity/over-explaining。
  Anthropic 公式: Opus はタスク複雑度を自分で判断し長さを決める、固定の簡潔既定は持たない。
- Claude Code には `type:"prompt"`（LLM-judge、既定 Haiku）/ `type:"agent"`（experimental）hook が
  ネイティブに存在（公式 hooks ドキュメントで確認）。
- **Stop hook の block = 継続（continue）であって再生成ではない**（公式確定）。`ok:false` の reason は
  次の指示として渡るだけで、表示済み応答は撤回できない。→ 喋りすぎ対策に Stop ゲートは構造的に不向き。
- 旧 `hooks/stop-scope-check.sh` を読解 + `scope-violation.log` の実発火を確認した結果、
  この hook は「block=再生成」を前提に `{decision:block, reason:"削除して再生成"}` を出していたが、
  実機構は「継続」。ログ上、発火時に Claude は違反応答を消せず「違反を確認しました…再生成します」と
  meta 前置き + やり直しを*追記*し、むしろ冗長化。加えて正規表現がコードブロック内の見出しや
  正当な `**ラベル**：` を誤検出。= 「効きがイマイチ」だった2つの機構的原因を実証。なお同 hook は
  既に `settings.json` の `Stop:[]` で未登録（dormant）。

**予防層の設計（Stop ゲートでなく予防側が機構的に正しいと検証で裏づけ）**：
- 編集 A: `output-styles/concise.md` にスコープ条項を追記（session-prompt 級・毎ターン無コスト、
  公式「否定形より肯定例」に従い肯定文で）。「聞かれたことだけに答える／資料提示には内容を一言で
  受け止め何をしてほしいか尋ねる／未要求の分析・批判・代案・推奨を足さない」。
- 編集 B: CLAUDE.md 批判的スタンス節を構造改修。スコープ限定を**節先頭の最優先ゲート**に格上げし、
  「問題点から先に述べる」を「判断・評価・レビューを求められたときのみ」の従属ルールに降格。
- UserPromptSubmit リマインダ hook は保留（リマインダは自己強制が弱く毎ターン context コスト、
  karpathy §2 に従い投機的に作らない）。

**empirical 検証（A/B、3 iteration）**：
- Iter1（対照=編集前 / 処置=編集後、ルーブリック開示）：全シナリオ両群 ○。だが**ルーブリックが
  desired behavior を実行者に教える汚染**を発見（skill「シナリオを修正に合わせてチューニング」の
  ルーブリック版）。S1/S2 の弁別は無効と判定。
- Iter2（汚染除去・ルーブリック非開示のブラインド）：**結論が覆る**。S1（コードを問いなしで提示）で
  対照は批判フル展開、処置も批判から開始し末尾に質問を足しただけ＝**未要求批判を止められず**。
  S2（事実質問）も両群が未要求の補足を付加。= append 型の編集 B は最強の事前傾向に埋もれ無効。
- Iter3（編集 B を構造改修してブラインド再試験）：**部分的成功**。S2 は `fix:` のみに改善（解決）、
  S3（レビュー依頼）は批判が正常発火し過剰補正＝黙り込みなし（維持）、S1 は出力が大幅短縮
  （脆弱性1点+1行修正+質問）したが**SQL インジェクションの指摘は依然保留せず**。

**判明したこと**：
1. プロンプト層は**最強の事前傾向（明白なセキュリティ欠陥の指摘）を完全には抑えられない**。
   「保留せよ」と最優先で指示しても SQLi は出る。これがプロンプト層の天井。
2. ゲートの格上げ＋従属化は S2/S3 に有効。スコープ限定を支配ルールにすると蛇足は消え、
   かつ判断要求時の批判は正常に通る（過剰補正しない）。
3. 機構的確実解は無い：Stop ゲートは撤回不可、UserPromptSubmit はソフト。ゼロ未要求批判を
   本気で求めるなら残るは「ユーザーが 1 行で割り込む」運用。
4. **方法論の教訓**：行動 A/B では、要件チェックリストを実行者に見せると desired behavior が
   漏れて両群とも正解する。スコープ系の検証はルーブリック非開示のブラインドで行うべき。

**決定**：構造改修版（編集 A + 構造改修した編集 B）を採用し、S1 残留はプロンプト層の天井として
受け入れて打ち切り。理由：append 版より厳密に良く（S2 解決・S1 短縮・S3 維持）、2 iteration で
S1 はプロンプトでは届かないと確認。実運用で体感評価する。

注記：subagent はロールプレイで実セッション挙動の完全な代理ではない。S1 残留の体感が許容外なら
別路線（運用での割り込み習慣）に切り替える。

**事後の文言修正**：検証は「明白な欠陥に気づいても、求められるまで指摘を保留する」で実施したが、
ユーザーから「保留する」が人間にも分かりにくいと指摘。抽象語をやめ動作で書く方針で
「欠陥に気づいても自分からは指摘せず、求められてから述べる」に置換してからコミット。
人が読んで引っかかる文言はエージェントにも伝わりにくい、という empirical-prompt-tuning の含意に沿う。



## 2026-05-30（続）: 批判的スタンス節を 3 層構成に再設計・ブラインド A/B 再検証

### 背景・方針転換
前回の予防層は「欠陥に気づいても自分からは指摘せず」と全面抑制していたが、ユーザーが「やっぱり欠陥は指摘してほしい。でないと対等な批判的相方の役目を果たせない」と再考。抑制軸を「指摘の有無」から「展開の有無」に付け替える方向へ。

### karpathy との整合確認
現行 CLAUDE.md と karpathy ガイドラインに硬い矛盾なしと確認。「欠陥に気づいても指摘しない」は karpathy §1（name confusion）・§3（mention, don't act）と矛盾しており、karpathy の型は「mention するが展開・実行はしない」。これに合わせて再設計。

### 3 層構成への再設計
「批判的スタンスは常時オン」を前提に、振る舞いを 3 層に分離:
- 土台（常時）: 相方姿勢・answer-first・重大欠陥の一言指摘・非迎合・淡々
- 増幅（判断・評価・レビュー等を求められたとき）: 問題先出し・反証探索・立場表明
- 抑制（資料提示・事実質問・実装依頼のとき）: 未要求の分析・代案・推奨を足さない・要否不明なら訊く
過程で「全面展開」「批判的振る舞い」等の不自然な造語・名詞化を排し、「以下の」の係り先も解消。

### ブラインド A/B（empirical-prompt-tuning）
共通ベース（concise + karpathy）に対し T=3 層節あり / C=節なし。実行ブラインド。
- iter1（自己採点）: S4（実装+SQLi）が唯一の弁別点。T は SQLi 一言指摘＋依頼分のみ実装、C は未要求の修正版を「推奨」提示。S1/S3 は T≈C（concise が S1、karpathy が S3 を既に賄う）。
- iter2（独立ブラインド判定者 + hold-out S2）: 自己採点が覆る。T-S1 失敗判定 — 欠陥が無いのに「事実誤りはありません」と自発報告＝未要求の先回り。S2/S4 は T 勝ち、S3 は引き分け。
- 修正: 土台 bullet を「気づいたときだけ指摘・無ければ何も言わない・『問題ない』と確認報告しない」に変更。再テストで S1 漏れ解消、S4 の指摘は維持。
- S4 は指摘どまり/指摘+修正の間でブレる（サンプリング由来、prompt 層の天井）。締めすぎると「重大欠陥は指摘してほしい」要件と衝突するため 80 点で確定。




## 2026-06-03: 批判的スタンス節の見出し構造を case A に統一・反証探索を削除・empirical 再検証

### 動機（構造の不整合指摘）
ユーザーが見出しの主従不整合を指摘。「土台（常時）」は役割名（主）＋タイミング（括弧）だが、「判断…を求められたとき（増幅）」はタイミング（主）＋役割名（括弧）で、条件分岐なのに軸が apple-to-apple でない。加えて導入文が「土台＝常時／増幅＝判断系」の二分法なのに本文は土台/増幅/抑制の三分法で、抑制のトリガーが導入文に書かれていない（ブロック数が 2 対 3）。

### 修正（case A: タイミングを主軸に統一 + 目的語明確化）
- 見出しを「常時（土台）／判断…のとき（批判を増幅）／資料提示…のとき（批判を抑制）」に統一。タイミングを主、役割名を括弧へ。
- 増幅/抑制の目的語が省略されていた点を解消。導入文を三分法に書き換え、「批判（問題先出し・立場表明の度合い）を増幅／抑制する」と目的語＝「批判」を明示。各見出しも「批判を増幅／抑制」に。

### 反証探索の削除
当初「自分の暫定結論に対する反証を最低 1 つ探す」と向きを固定する修正を入れたが、ユーザーが「重箱の隅の間違いを指摘しだして話が前に進まない」として bullet ごと削除を指示。導入文の目的語リストからも「反証探索」を除去（消し残すと宙に浮くため）。「私はしばしば間違う。前提を能動的に疑う。」の姿勢文は残置。

### empirical-prompt-tuning（2 イテレーション、収束）
新規 subagent に節を渡し挙動観察。critical = モード判定。
- iter1（案 A 改訂版、A 増幅/B 抑制/C 混在）: 全 ○、目的語＝批判を全員特定、C は「向きだけ逆の同一つまみ」と正解。新出不明瞭点は「反証の向きが未規定」（A・C が独立指摘、retry 源）と「土台×増幅/抑制の干渉」。
- 反証行を削除（上記）。
- iter2（D 抑制×土台欠陥/E 増幅×土台で問題薄/F 増幅中央値のリグレッション）: 全 ○。iter1 残課題の干渉はテキスト修正なしで解決 — D は「土台の一言指摘と抑制が抑える網羅レビューは別物」、E は「土台の沈黙ルールは未要求指摘の規定、要求された判断には立場表明義務が残る」と正しく読解。反証削除によるリグレッションなし（F が問題先出し＋立場表明で 90%）。

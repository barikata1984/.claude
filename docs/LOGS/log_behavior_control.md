# Log: 挙動制御フレームワーク

## 2026-05-04: scope expansion 抑制と CLAUDE.md 剪定

### 問題診断

ユーザーから挙動の指摘：「A という質問に答える際、Claude が B/C
という関連話題を勝手に提示し、自分の希望（B について話したい等）まで
表明する」型の scope expansion。加えて「過度な迎合（sycophancy）」も
抑制したいが、まずは前者から対処する方針。

文献調査の結果（WebSearch / WebFetch）：

- **Excessive Agency** (OWASP LLM06:2025) — ツール権限の脆弱性なので、
  この現象を直接記述する用語ではない（誤用注意）
- **Verbosity Compensation** (arXiv:2411.07858) — 不確実性下での冗長応答。
  scope 拡張とは別概念
- **Sycophancy** (arXiv:2310.13548, Anthropic) — ユーザー見解への迎合
- **Verbosity Bias** (arXiv:2310.10076, Saito et al.) — preference labeling
  でのバイアス
- **Show and Tell** (arXiv:2511.13972) — directive 介入で 56% 削減、
  directive + 例示で 70% 削減を実証

「scope expansion」を直接命名する標準用語は存在せず、Anthropic の
[claude-personal-guidance](https://www.anthropic.com/research/claude-personal-guidance)
が報告する sycophancy の親戚として扱える。`anthropics/claude-code` issue
[#27147](https://github.com/anthropics/claude-code/issues/27147) が
「Claude が research answer の後に勝手に write アクションを取る」現象を
記述しており、最も直接的な関連 issue。

### 対策の階層（layered defense）

1. **CLAUDE.md directive**（予防、~56% 効果）
2. **Stop hook**（決定論的検査、100% 効果だがパターンマッチ精度依存）
3. **将来オプション**：UserPromptSubmit hook、subagent systemPrompt override 等

### Stop hook 実装

- 新規 `~/.claude/hooks/stop-scope-check.sh`（block mode）
  - `^#{2,}\s+.*(なぜ|結論|推奨|次のステップ|要点|まとめ|補足|例|比較|方針)`
    にマッチする見出しを検出
  - 検出時は `{"decision": "block", "reason": "..."}` を出力して再生成を強制
  - `stop_hook_active=true` 時は無限ループ防止のためパススルー
  - 違反ログは `~/.claude/hooks/scope-violation.log` に記録
- 既存 `~/.claude/hooks/stop-taxonomy-check.sh` を warn-only から block へ昇格
  - `stop_hook_active` チェックを追加
  - 検出時のログ記録を維持しつつ block JSON を出力
- `settings.json` の Stop 配列に scope-check を並列登録（同一 entry に追加）

### 設計判断：分離 vs 統合

scope-check と taxonomy-check は別スクリプトとして分離。理由：
- 検出対象が直交（taxonomy ハルシネーション vs 未要求サブセクション）
- 既存 taxonomy-check の試運用観察（ログ蓄積）を撹乱しない
- ログを分離（`taxonomy-warn.log` / `scope-violation.log`）して個別に偽陽性率
  評価が可能
- 重複する transcript parsing logic（~40 行の jq pipeline）はコピペ容認、
  3 つ目の hook が必要になった時点で共通関数に抽出（YAGNI）

## 2026-05-22 和文タイポgrafィ正規化フック実装 + 応答制御の concise スタイル移行

### 和文タイポグラフィ正規化フック（新規実装）

英文・和文混在の技術文書で、半角約物スタイル（理工系スタイル）を `.md` 生成時に
自動適用するフックを実装。設計判断は以下。

**二層構成**：判断を要する規則（ダッシュ使い分け・和欧境界スペースの語彙的例外）は
プロンプト（`rules/japanese_style.md`）、機械判定できる規則（全角約物→半角）は
決定論的フックに分離。フックはチャット応答には効かず Write/Edit したファイルのみ対象。

**変換規約（全角約物のみ変換・ASCII 不可侵）**：
- `。`→`. ` / `、`→`, ` / `！`→`! ` / `？`→`? ` / `：`→`: `
- `「」『』`→`"" ''`、`（）`→`()`、`〜`・`・` は温存
- ASCII の `.` `,` は触らないため小数 `3.14`・略語 `e.g.`・URL・コードは自動保護
- ダッシュ: em dash 禁止は生成時、en dash は範囲・複合語限定。フックは検出して警告のみ
- 数字スペース: 直後が漢字かな=詰める / ラテン文字=空ける / `%`・`°`=詰める（生成時規則）

**実装手段の確定経緯**：当初 markdown-it-py の AST で変換を計画したが、実測で
(1) インライントークンが元ソースの文字位置を持たない (map=None)、(2) render() が
HTML を返し Markdown に戻せない、ことが判明。AST 経由を断念し**正規表現マスク方式**に
変更。保護対象（フロントマター・フェンス/インラインコード・数式・LaTeX・リンク/URL）を
順にマスク→残りを変換→復元。

**成果物**：
- `~/.claude/hooks/format_ja_typography.py`（マスク方式エンジン、フェイルセーフ・冪等）
- `~/.claude/hooks/test_format_ja_typography.py`（24 ケース + 冪等性 + 警告検出、全通過）
- `~/.claude/hooks/format-ja-typography.sh`（`.md` のみ・`~/.claude/**` 除外・
  opt-out マーカー `<!-- no-ja-typography -->` 対応）
- `settings.json` PostToolUse に登録（ユーザーが手動登録、フック登録は分類器がブロック）
- `rules/japanese_style.md` に「タイポグラフィ（半角約物スタイル）」節を追記

**適用範囲**：global（全 .md）。PostToolUse は編集したファイルにのみ発火するため、
触らないノートは無傷。LaTeX ソースは opt-out マーカーで除外。

### 応答制御：scope-check 路線から concise 出力スタイルへ移行

**検証で判明した事実**：
- 「1-3 文 / preamble 禁止」という固定の組み込み既定は**存在しない**（公式 Claude Code
  ベストプラクティス全文を確認）。Opus 4.7 は複雑性に応じて応答長を調整し固定既定を
  持たない設計（Anthropic 公式プロンプトガイドで確認）。以前「組み込み既定」と呼んだ
  のは非公式ミラー由来の誤りで撤回。
- 公式は「冗長削減はプロンプトで明示せよ、否定形より肯定例が有効」と明記。
- Claude Code には**出力スタイル**機能あり（`~/.claude/output-styles/*.md`、

  注入）より、振る舞い定義にはこちらが設計上適切。ブラウザの `<userStyle>` は browser-only。

**真因の特定**：本セッション中の冗長な応答（未要求の太字疑似見出しの濫用・反復）の真因は
規則の不在ではなく、既存「応答スコープ規約」をエージェント側が遵守しない**遵守不全**。
規則は自己強制しないため、重複する新規則を足しても効果は限定的（セッション中に実演された）。

**方針転換の経緯**：
1. 「応答密度規約」新節の追加を検討 → 組み込み既定・既存スコープ規約と重複し不要と判断

3. ブラウザ版 concise スタイル文面の存在を受け、出力スタイル機能での実装に方針変更
4. concise 単体の効果を純粋に観察するため、**独自二条項（スコープ制約・反復禁止）を

**成果物**：

### 設計判断：長さ上限フックの見送り

反復抑制の手段として (1) プロンプト規則 (2) 長さ上限 Stop フック (3) ユーザーの即時指摘、
の三段を検討。長さ閾値は恣意的で試行錯誤コストが高いため、まずプロンプト/スタイル単体で
運用し、効果不足なら閾値フックを追加する段階導入を選択。情報密度 = 情報量/文量のうち
分子（情報量）と「言い換え反復」は semantic で機械判定不可、分母（文量）のみ決定論的に
制約可能、という分析に基づく。

## 2026-05-26（続 2）: 情報の確実性ポリシーの内部再構成 — 形式の単一基準

**問題**：前エントリで導入した番号フロー（トリガー → 検証 → 報告 → 開示）は、(1) 番号
リストが「線形手順」を示唆するのに中身は条件・手順・例外が混在、(2) フローと配下の
bullet で同内容を二重化、(3) 「段階 N の詳細」という従属ラベルが知識ソース明示ルールを
不当に従属させる、という欠陥を持っていた。修正のたびに番号 ↔ 中点 ↔ 括弧 ↔ 散文を
場当たりに入れ替える継ぎ接ぎ状態に陥った。

**ベストプラクティス調査**（WebSearch）：
- Anthropic 公式の「散文推奨・リスト回避」は Claude の*出力フォーマット*向けの指針で、
  *入力指示*（CLAUDE.md）には適用されない。両者の混同が誤判断の一因だった。
- 指示遵守には条件 → アクション（if-then）の明示と、独立ガイドラインの箇条書き分離が
  有利。ただし分離が効くのは独立項目のときで、データ依存の連鎖を分離すると依存が
  消える。最小構造で目的を達成する（Anthropic）。

**確立した単一基準**：形式を中身の論理構造に従わせる。
- 連続した 1 つの規範（手段と結果の依存連鎖）→ 散文。番号で割らない。
- 独立して「どれか該当で発火」する複数条件 → 箇条書き。
- 規範本体は本文に置き、括弧は飛ばしても本筋が成立する真の補足（検証手段の例など）
  のみに使う。規範を括弧に入れると遵守優先度が下がる（従属化の誤り）。

**変更内容**：
- フロー番号（1-4）と配下 bullet を全廃。`### 検証による解消`（散文の規範 + 発火条件の
  箇条書き 6 項目）と `### 知識ソースの明示`（散文の規範 + 付与条件の箇条書き 3 項目）の
  2 サブセクションに再構成。
- 旧「WebSearch / WebFetch 必須トリガー」見出しを廃し、検証を強制する条件として
  「検証による解消」配下の箇条書きに統合。節内参照も「『検証による解消』の必須
  トリガー」に調整。
- 旧フロー段階 4（残余の仮説明示）は知識ソースの明示の付与条件 3 つ目に吸収。
- 「段階 N の詳細」という従属ラベルを全廃し、2 サブセクションを対等化。

**設計判断**：散文／箇条書きの使い分けを「連続規範か／独立発火条件か」という単一基準で
全ブロック説明できる状態にし、継ぎ接ぎを解消した。

## 2026-05-27: karpathy-guidelines プラグインの常時ロード可否評価 — 不採用

**背景**：インストール済みプラグイン `andrej-karpathy-skills@karpathy-skills`（skill
`karpathy-guidelines`、Andrej Karpathy の LLM コーディング失敗観察に基づく 4 原則）を、
毎セッション自動ロードして CLAUDE.md と併用すべきか評価した。

**トピック網羅マトリクス**：karpathy の 4 原則（Think Before Coding / Simplicity First /
Surgical Changes / Goal-Driven Execution）はすべて CLAUDE.md 既存セクションに対応物を持ち、
karpathy 固有の新規トピックはゼロ。正味の新規貢献は Surgical Changes の「既存スタイル尊重・
無関係な隣接コードを触らない」というコード編集固有の具体性のみ。

**競合分析**：唯一の真の競合は「プロンプト側の明確化質問の閾値」。karpathy 項目 1 は
「迷えば聞け・選択肢を提示せよ」、応答スコープ規約は「真に曖昧な時のみ 1 つ・列挙して
回避するな」で逆向き。両者とも対象は*ユーザーのプロンプト*であり、対象の混同ではなく
同一対象上の閾値衝突である。これは情報の確実性ポリシー（対象は*エージェントが集めた/
創発した情報*）とは別物で、後者は karpathy に対応物が無く、karpathy 主軸にすると脱落する
（競合ではなく欠落）。

**機構確認**（claude-code-guide エージェント経由でドキュメント検証）：
- skill 本体は遅延ロードで、常時ロードを強制する設定は存在しない。
- プラグイン root の CLAUDE.md は自動ロードされない（人間向けドキュメント扱い）。
- 毎セッション確実に注入する手段は、ユーザー CLAUDE.md 直書き / `@path` インポート
  （起動時に全文展開、最大 5 段ネスト、初回承認ダイアログ）/ SessionStart フックの
  `additionalContext` 出力の 3 つ。

## 2026-05-30: 過剰応答（未要求出力・喋りすぎ）への予防層導入と検証

**問題**：批判的スタンス節の導入後、エージェントが「先走って依頼していない情報を出力し、結局
何を言っているか分からない」挙動が頻発（資料を貼っただけで未要求の批判レビューを始める等）。
真因は、批判的スタンス節の「問題点・リスクから先に述べる」が、スコープ・ガード不在のまま
資料提示・事実質問にも適用されたこと。これは過去に削除した「応答スコープ規約」と同型の課題。

**機構の検証（推測でなく一次情報で確定）**：
- 既知の名称は scope creep（karpathy が Claude Code 最頻失敗と指摘）+ verbosity/over-explaining。
  Anthropic 公式: Opus はタスク複雑度を自分で判断し長さを決める、固定の簡潔既定は持たない。
- Claude Code には `type:"prompt"`（LLM-judge、既定 Haiku）/ `type:"agent"`（experimental）hook が
  ネイティブに存在（公式 hooks ドキュメントで確認）。
- **Stop hook の block = 継続（continue）であって再生成ではない**（公式確定）。`ok:false` の reason は
  次の指示として渡るだけで、表示済み応答は撤回できない。→ 喋りすぎ対策に Stop ゲートは構造的に不向き。
- 旧 `hooks/stop-scope-check.sh` を読解 + `scope-violation.log` の実発火を確認した結果、
  この hook は「block=再生成」を前提に `{decision:block, reason:"削除して再生成"}` を出していたが、
  実機構は「継続」。ログ上、発火時に Claude は違反応答を消せず「違反を確認しました…再生成します」と
  meta 前置き + やり直しを*追記*し、むしろ冗長化。加えて正規表現がコードブロック内の見出しや
  正当な `**ラベル**：` を誤検出。= 「効きがイマイチ」だった2つの機構的原因を実証。なお同 hook は
  既に `settings.json` の `Stop:[]` で未登録（dormant）。

**予防層の設計（Stop ゲートでなく予防側が機構的に正しいと検証で裏づけ）**：
- 編集 A: `output-styles/concise.md` にスコープ条項を追記（session-prompt 級・毎ターン無コスト、
  公式「否定形より肯定例」に従い肯定文で）。「聞かれたことだけに答える／資料提示には内容を一言で
  受け止め何をしてほしいか尋ねる／未要求の分析・批判・代案・推奨を足さない」。
- 編集 B: CLAUDE.md 批判的スタンス節を構造改修。スコープ限定を**節先頭の最優先ゲート**に格上げし、
  「問題点から先に述べる」を「判断・評価・レビューを求められたときのみ」の従属ルールに降格。
- UserPromptSubmit リマインダ hook は保留（リマインダは自己強制が弱く毎ターン context コスト、
  karpathy §2 に従い投機的に作らない）。

**empirical 検証（A/B、3 iteration）**：
- Iter1（対照=編集前 / 処置=編集後、ルーブリック開示）：全シナリオ両群 ○。だが**ルーブリックが
  desired behavior を実行者に教える汚染**を発見（skill「シナリオを修正に合わせてチューニング」の
  ルーブリック版）。S1/S2 の弁別は無効と判定。
- Iter2（汚染除去・ルーブリック非開示のブラインド）：**結論が覆る**。S1（コードを問いなしで提示）で
  対照は批判フル展開、処置も批判から開始し末尾に質問を足しただけ＝**未要求批判を止められず**。
  S2（事実質問）も両群が未要求の補足を付加。= append 型の編集 B は最強の事前傾向に埋もれ無効。
- Iter3（編集 B を構造改修してブラインド再試験）：**部分的成功**。S2 は `fix:` のみに改善（解決）、
  S3（レビュー依頼）は批判が正常発火し過剰補正＝黙り込みなし（維持）、S1 は出力が大幅短縮
  （脆弱性1点+1行修正+質問）したが**SQL インジェクションの指摘は依然保留せず**。

**判明したこと**：
1. プロンプト層は**最強の事前傾向（明白なセキュリティ欠陥の指摘）を完全には抑えられない**。
   「保留せよ」と最優先で指示しても SQLi は出る。これがプロンプト層の天井。
2. ゲートの格上げ＋従属化は S2/S3 に有効。スコープ限定を支配ルールにすると蛇足は消え、
   かつ判断要求時の批判は正常に通る（過剰補正しない）。
3. 機構的確実解は無い：Stop ゲートは撤回不可、UserPromptSubmit はソフト。ゼロ未要求批判を
   本気で求めるなら残るは「ユーザーが 1 行で割り込む」運用。
4. **方法論の教訓**：行動 A/B では、要件チェックリストを実行者に見せると desired behavior が
   漏れて両群とも正解する。スコープ系の検証はルーブリック非開示のブラインドで行うべき。

**決定**：構造改修版（編集 A + 構造改修した編集 B）を採用し、S1 残留はプロンプト層の天井として
受け入れて打ち切り。理由：append 版より厳密に良く（S2 解決・S1 短縮・S3 維持）、2 iteration で
S1 はプロンプトでは届かないと確認。実運用で体感評価する。

注記：subagent はロールプレイで実セッション挙動の完全な代理ではない。S1 残留の体感が許容外なら
別路線（運用での割り込み習慣）に切り替える。

**事後の文言修正**：検証は「明白な欠陥に気づいても、求められるまで指摘を保留する」で実施したが、
ユーザーから「保留する」が人間にも分かりにくいと指摘。抽象語をやめ動作で書く方針で
「欠陥に気づいても自分からは指摘せず、求められてから述べる」に置換してからコミット。
人が読んで引っかかる文言はエージェントにも伝わりにくい、という empirical-prompt-tuning の含意に沿う。



## 2026-05-30（続）: 批判的スタンス節を 3 層構成に再設計・ブラインド A/B 再検証

### 背景・方針転換
前回の予防層は「欠陥に気づいても自分からは指摘せず」と全面抑制していたが、ユーザーが「やっぱり欠陥は指摘してほしい。でないと対等な批判的相方の役目を果たせない」と再考。抑制軸を「指摘の有無」から「展開の有無」に付け替える方向へ。

### karpathy との整合確認
現行 CLAUDE.md と karpathy ガイドラインに硬い矛盾なしと確認。「欠陥に気づいても指摘しない」は karpathy §1（name confusion）・§3（mention, don't act）と矛盾しており、karpathy の型は「mention するが展開・実行はしない」。これに合わせて再設計。

### 3 層構成への再設計
「批判的スタンスは常時オン」を前提に、振る舞いを 3 層に分離:
- 土台（常時）: 相方姿勢・answer-first・重大欠陥の一言指摘・非迎合・淡々
- 増幅（判断・評価・レビュー等を求められたとき）: 問題先出し・反証探索・立場表明
- 抑制（資料提示・事実質問・実装依頼のとき）: 未要求の分析・代案・推奨を足さない・要否不明なら訊く
過程で「全面展開」「批判的振る舞い」等の不自然な造語・名詞化を排し、「以下の」の係り先も解消。

### ブラインド A/B（empirical-prompt-tuning）
共通ベース（concise + karpathy）に対し T=3 層節あり / C=節なし。実行ブラインド。
- iter1（自己採点）: S4（実装+SQLi）が唯一の弁別点。T は SQLi 一言指摘＋依頼分のみ実装、C は未要求の修正版を「推奨」提示。S1/S3 は T≈C（concise が S1、karpathy が S3 を既に賄う）。
- iter2（独立ブラインド判定者 + hold-out S2）: 自己採点が覆る。T-S1 失敗判定 — 欠陥が無いのに「事実誤りはありません」と自発報告＝未要求の先回り。S2/S4 は T 勝ち、S3 は引き分け。
- 修正: 土台 bullet を「気づいたときだけ指摘・無ければ何も言わない・『問題ない』と確認報告しない」に変更。再テストで S1 漏れ解消、S4 の指摘は維持。
- S4 は指摘どまり/指摘+修正の間でブレる（サンプリング由来、prompt 層の天井）。締めすぎると「重大欠陥は指摘してほしい」要件と衝突するため 80 点で確定。



## 2026-05-30（続々）: under-retrieval 対策「外部事実の検証」節を新設（段階 1）

### 問題の切り分け（hallucination ではなく under-retrieval）
調査・デバッグ依頼で、(a) 検索すべき外部知識を取りに行かず内部知識だけで答える、(b) 検索しても浅く取り残す、という困りごと。当初 hallucination 枠で調査を始めたが、議論の結果 under-retrieval（外部知識導入のサボり = 行動選択の失敗）と切り分けた。hallucination（生成の捏造）はその下流の二次的帰結であって別物。対策の重心は「検索を起動・深化させる」に置き、confidence 宣言・claim 撤回等の hallucination 対策一式は不要と判断。

### 批判的スタンスとの分離
当初 CLAUDE.md の批判的スタンス節に相乗りさせようとしたが、規律のカテゴリが別（批判的スタンス = 対ユーザーの態度 / 外部事実の検証 = 対・事実の情報取得衛生）。混ぜると節の凝集性が壊れ、empirical でも両者の A/B が干渉するため、独立節として新設。共通の上流は karpathy §1。

### 既存実装の調査
- Anthropic 公式「Reduce hallucinations」: 「I don't know 許可」「claim 撤回」「外部知識制限」。
- Aedelon/claude-code-anti-hallucination: Haiku 4.5 が prompt を VOLATILE/NONE に二値分類し、VOLATILE のときだけ UserPromptSubmit hook で検証プロトコルを inline 注入（NONE は素通り）。自己申告 F1 = 0.988。「ゲート（分類）とプロトコル（実行）の責務分離」が最大の収穫 = over-trigger 回避の鍵。
- Aedelon/claude-code-blueprint: research-protocol skill（SCOPE → SEARCH → EVALUATE → SYNTHESIZE → CITE → ASSESS、source hierarchy）。失敗 (b) の深さ対策は既存 deep-research でカバー可。

### 設計判断
- 強制はハーネスでは原理的に不可能（検索する/しないの判断はモデル内部で起き、PreToolUse 等では観測不可）。
- Stop hook の事後ゲート（検索ゼロでの終了を差し戻す）は却下: スクリプトは意味判定できず、全質問に検索を強要して over-response を招く（CLAUDE.md の抑制方針と衝突）。
- 常駐 Haiku デーモンは個人環境に過剰・MCP 依存・F1 が自己申告のため不採用。思想（分類 → 注入の責務分離）のみ借用。
- 段階導入を採用: 段階 1 = CLAUDE.md に薄い指示節、段階 2 = empirical 測定、段階 3 = 忘却が残れば軽量 UserPromptSubmit hook（正規表現分類）。デーモン・hallucination 対策一式は持ち込まない。

### 段階 1 の実装
CLAUDE.md に新節「外部事実の検証」を追加（実験コメント付き）。トリガーを「外部事実を述べるとき」に限定（内部知識で確実なもの・trivial は素通り）、確認できなければ「内部知識・未確認」と明示、深さの要る調査は deep-research に委譲。

### empirical-prompt-tuning（収束）
5 シナリオ、いずれも新規 subagent に節を渡して挙動を観察:
- A（バージョン固有 API、検索すべき）: WebSearch/WebFetch 起動・出典明示・未確認明示。○
- B（再帰関数、一般概念）: 検索ゼロで即答。○

### 限界と残課題
- empirical は「指示の明瞭さ（白紙の実行者が場合分けできるか）」を測れたが、「実運用の遵守率（CLAUDE.md に書いてあるのにコスト判断で従わないか）」は測れない。これが最大リスクで、実運用観察待ち。
- 全シナリオで subagent が「補足を足すか、抑制節と緊張する」と裁量補完で迷った = 外部事実の検証節ではなく批判的スタンスの抑制節との相互作用。別途観察。

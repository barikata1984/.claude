## 全体方針

研究計画書には大きく二つの代表的な型がある。両者は対立するものではなく、**目的に応じて組み合わせる**のが実用的である。

| 型 | 特徴 | 適した状況 |
|---|---|---|
| Heilmeier 型 | 質問形式で価値・実現性を評価 | 助成申請、社内提案、研究の意義の検証 |
| ML 論文構造型 | 論文セクションを未来形で書く | 学位論文、論文化を意識した計画 |

**重要な原則**: 研究計画書は「自分が何をやりたいか」を整理する文書ではなく、「**読み手にこの研究の価値と実現可能性を納得させる**」ための文書である。読み手の視点で書く。

## Heilmeier 教義 (Heilmeier Catechism)

DARPA の元ディレクター George Heilmeier が作成した、研究提案を評価するための質問群。1975 年から使われており、現在も多くの助成機関の評価軸に組み込まれている。

### 8 つの質問

1. **何をやろうとしているのか?** 専門用語を一切使わずに目的を表現する
2. **現在はどうやられているか? 現状の手法の限界は?**
3. **アプローチの新規性は何か? なぜ成功すると思うのか?**
4. **誰が気にするのか?** ステークホルダーの特定
5. **成功したらどんな違いを生むのか?** 期待される影響
6. **リスクと見返りは何か?**
7. **コストはいくらかかるか?**
8. **どれくらい時間がかかるか? 中間チェックと最終評価は何か?**

### 各質問の意図

**質問 1: 専門用語禁止の意味**
これは単なる読みやすさではない。**専門用語抜きで説明できなければ、その研究は本質を掴めていない**という診断ツール。家族や指導教員以外の人に説明して理解されるか試すことが推奨される。

**質問 2: 現状把握の徹底**
「既存研究は不十分」と漠然と書くのではなく、「どの論文の何の手法が、どういう設定で、どこまで達成しているか」を具体的に書く。これができないなら、その研究は新規性を主張できる立場にない。

**質問 3: 成功の根拠**
「やってみないと分からない」は許容されない。**なぜ自分のアプローチが成功すると信じるのか**、理論的根拠か、予備実験か、類似手法の成功実績か、何かしらの根拠を示す。

**質問 4: 「誰が気にするのか?」の重み**
研究の社会的・学術的意義を強制的に問う。「面白い」だけの研究は支援に値しない、という思想が背景にある。「興味深い研究の全てが重要なわけではない」というメッセージ。

**質問 5: インパクトの具体化**
「分野に貢献する」ではなく、具体的に何が変わるかを書く。例: 「リアルタイム制御要件を満たす最初のJEPA計画器となり、シミュレーション中心の研究領域が実機ロボティクスに移行する基盤になる」。

**質問 6-8: 実現可能性**
野心と現実のバランス。リスクなしの計画は野心が低く、リスクだらけの計画は実現可能性がない。**段階的なマイルストーン**で部分的な成功も担保する設計が望ましい。

### Heilmeier 型のテンプレート

```markdown
## 1. 何をやろうとしているのか
[専門用語抜きで 2-3 段落]

## 2. 現状の手法と限界
[既存手法を具体名で挙げ、それぞれの限界を定量的に示す]

## 3. アプローチの新規性
[何が新しいか、なぜ機能すると考えるか]

## 4. 関連する利害関係者
[研究コミュニティ、産業、社会のどこに影響するか]

## 5. 期待されるインパクト
[成功時の具体的な変化]

## 6. リスクと対応
[失敗シナリオと、それぞれへの対応策]

## 7. リソース
[人員、計算機、データ、その他]

## 8. スケジュールと評価
[フェーズ分け、各フェーズの中間検証]
```

## ML 論文構造を計画書に適用する型

NeurIPS / ICML / ICLR の論文構造を、まだ実験していない計画書に適用する手法。**論文化への道筋を最初から意識**できる利点がある。

### 標準的なセクション構成

```
1. Abstract / Executive Summary
2. Introduction
3. Related Work
4. Background / Preliminaries
5. Method / Proposed Approach
6. Experiments (計画として記述)
7. Discussion / Limitations
8. Conclusion / Future Work
```

### 各セクションの書き方

**Abstract / Executive Summary** (1 段落)
- 研究の全体像を 150-300 語で要約
- 何を、なぜ、どのように、何を期待するか
- 読み手はここで読み続けるかを決めるので、最重要

**Introduction**
推奨される 4 段落構成:
1. **問題の重要性**: 取り組む課題が広く重要であることを示す
2. **既存研究の限界**: 現状の手法が何故不十分かを具体的に
3. **本研究の貢献**: 提案手法と新規性を簡潔に
4. **論文の構成**: 以降のセクション概要

**Related Work**
三つの軸で整理する:
- 同じ問題を別アプローチで解いている研究
- 似たアプローチを別問題で使っている研究
- 自分の研究に直接組み込む基盤研究

**重要**: 単なる文献羅列にしない。各研究を引用するときに「**なぜそれが本研究と関連し、本研究はそれと何が違うか**」を一文で示す。

**Background / Preliminaries**
- 必要な記法・定義の導入
- 既存手法の説明(本研究が依拠するもの)
- **読み手が論文単独で内容を追えるよう**にする

**Method / Proposed Approach**
- 提案手法の詳細
- 数式と図で形式化
- **なぜそれが機能するかの理論的根拠**を含める
- 「動く」だけでなく「動く理由」を説明する

**Experiments (計画として)**
研究計画書では実際の結果はないので、以下を書く:
- 検証する仮説の明示(**反証可能な形で**)
- ベンチマーク・データセット
- ベースライン手法
- 評価指標
- アブレーション計画(各コンポーネントの寄与を切り分ける設計)

**Discussion / Limitations**
- 期待される結果の解釈
- 想定される手法の限界
- 失敗シナリオ
- ネガティブ結果の場合の対応

**Conclusion / Future Work**
- 研究の意義の再確認
- 本研究で取り組まないが関連する将来の方向性

## 統合的な構成テンプレート

実用的には、Heilmeier 型と ML 論文構造型を **3 部構成** で統合するのが効果的。

```markdown
# タイトル

## 第 1 部: 動機 (Heilmeier 1-5 に対応)
- 1.1 平易な要約 (専門用語抜き)
- 1.2 問題提起
- 1.3 既存研究の現状と限界
- 1.4 提案手法の新規性
- 1.5 期待される影響

## 第 2 部: 技術的内容 (ML 論文構造型)
- 2.1 背景知識
- 2.2 関連研究
- 2.3 提案手法
- 2.4 理論的解析

## 第 3 部: 検証計画 (Heilmeier 6-9 に対応)
- 3.1 検証する仮説
- 3.2 実験設計
- 3.3 マイルストーンと中間検証
- 3.4 リスクと対応
- 3.5 想定されるネガティブ結果
- 3.6 成功基準

## 第 4 部: 補足
- 4.1 予備実験結果
- 4.2 今後の展望
- 4.3 参考文献
```

この構成の利点:
- **動機 → 技術 → 検証** の論理的な流れ
- 第 1 部で価値を、第 2 部で実現方法を、第 3 部で検証可能性を担保
- そのまま論文ドラフトに転換可能

## 実践的な書き方の指針

### 仮説は反証可能な形で書く

悪い例:
> 提案手法は計画速度を向上させる

良い例:
> **仮説 H1**: 提案手法は Push-T ベンチマークで、LeWM (CEM ベース) と同等の成功率 (90% 以上) を維持しつつ、計画時間を 1 秒/プランから 0.1 秒/プラン以下に短縮する。

ポイント:
- **数値目標**: 「向上する」ではなく「X% 以上」
- **比較対象**: 何と比較するかを明示
- **条件**: どの環境・設定で検証するか
- **反証可能**: 達成できない場合に明確に判定できる

### マイルストーンには中間検証を含める

各フェーズで「**何が確認できれば次に進む価値があるか**」を明示する。これがないと、計画全体が「最後まで全てうまくいく前提」になり、リスク管理ができない。

例:
```markdown
### Phase 2: 簡易プロトタイプ実装 (週 3-5)

**作業内容**:
- API を拡張し、簡易プランナを実装
- ベンチマーク環境で動作確認

**中間検証 (このフェーズの成功基準)**:
- 計画時間が CEM の 1/10 以下になっているか
- 成功率が CEM の 50% 以上維持されているか

**失敗時の対応**:
- 計画時間が短縮されない → 実装のボトルネックを分析、CEM 除去の効果を再検証
- 成功率が極端に低い → ハイブリッド設計 (FM + 短い CEM 補正) に切り替え
```

### リスクは正直に書き出す

リスクを書かない計画書は逆に信用されない。**研究には不確実性があるのが当然**で、それを把握しているかが評価される。

リスク表のテンプレート:

| リスク | 確率 | 影響 | 対応 |
|---|---|---|---|
| 主要な失敗シナリオ | 低/中/高 | 低/中/高 | 具体的な対応策 |

### ネガティブ結果のシナリオを書く

「うまくいかなかった場合、研究はどう着地するか」を計画段階で考える。これがあると:
- 最悪のケースでも論文化できる道筋が見える
- 仮説が外れても、なぜ外れたかが価値ある発見になる設計ができる

例:
```markdown
**ネガティブシナリオ**: 提案手法 X が経験的に効果を示さない場合
→ 「期待された効果が観測されなかった」自体を実証的発見として論文化。
   実装の正しさを担保するため、Y, Z のアブレーションで X の不寄与を切り分ける。
```

### 成功基準を段階化する

「全か無か」ではなく、達成度に応じた段階的成功基準を設定する。

```markdown
**最低限の成功 (論文化可能)**: 
- 主要ベンチマークで既存手法と同等以上

**標準的な成功 (中堅会議に投稿可能)**:
- 上記に加え、複数環境での一貫した改善

**最大の成功 (Top-tier 会議で強い貢献)**:
- 上記に加え、新しい応用領域への展開
```

### ジャーゴン削減の実践

専門用語を削るプロセスは、研究の本質を浮かび上がらせる。

手順:
1. 一度全力で専門用語ありで書く
2. 一度通読して、専門用語を全てマーク
3. それぞれを「専門外の人にも分かる表現」に置き換えられるか試す
4. 置き換えで意味が崩れる用語だけ残す

例:
- ❌ 「JEPA の SIGReg 制約下で flow matching の score decomposition を行う」
- ⭕ 「学習した世界の内部表現が持つ統計的性質を活用して、計画の生成を高速化する」

第 1 部の冒頭は後者で書き、第 2 部以降で専門用語を導入する。

## よくある失敗パターン

研究計画書でよく見られる失敗とその回避法。

### 失敗 1: 既存研究の調査が浅い

症状:
- 「既存研究は不十分」と漠然と書く
- 引用文献が少ない、または分野の代表的論文が抜けている
- 自分の手法を既存研究と並べて配置できていない

回避法:
- **少なくとも 30-50 本の関連文献**を読み込む
- 関連研究を表形式で整理し、各軸での比較を可視化
- 引用ごとに「なぜ関連するか」を一文で書く

### 失敗 2: 動機が弱い

症状:
- 「面白そう」「やってみたい」が動機の中心
- 解決される問題が抽象的
- 誰が困っているのかが不明確

回避法:
- Heilmeier 質問 4「誰が気にするのか?」に具体的に答える
- 問題が解決された場合の具体的な変化を示す
- 数値で示せるなら示す(例: 「制御周波数が 1Hz から 100Hz に上がる」)

### 失敗 3: 実験計画が抽象的

症状:
- 「ベンチマークで評価する」とだけ書く
- ベースラインが不明
- 何を持って成功とするかが書かれていない

回避法:
- ベンチマーク名、データセット名、ベースライン名を具体的に書く
- 評価指標と数値目標を明示
- アブレーション計画で各要素の寄与を切り分ける設計

### 失敗 4: 楽観的すぎる

症状:
- リスクが書かれていない
- 全てのフェーズが順調に進む前提
- 失敗シナリオへの対応がない

回避法:
- リスク表を必ず含める
- 各フェーズに中間検証点を設ける
- 想定されるネガティブ結果と、その場合の対応を書く

### 失敗 5: 範囲が広すぎる

症状:
- 一つの計画書で複数の独立した研究を含もうとする
- 16 週で 5 つの会議論文分の研究をしようとする

回避法:
- 一つの計画書 = 一つの主張
- 副次的なアイデアは「Future Work」に回す
- 各フェーズの所要時間を現実的に見積もる(過去の類似研究を参考に)

### 失敗 6: 自分の研究の弱点を隠す

症状:
- 都合の悪いベースラインを比較から除外する
- 限界を Limitations セクションで触れない
- 失敗時の対応を書かない

回避法:
- 弱点や限界を **積極的に書き出す**
- 査読者が指摘するであろう批判を先回りして回答
- ネガティブな可能性も含めて書くことで、計画全体の信頼性が上がる

## 読み手別の調整

研究計画書の読み手によって、強調すべき点や使う言葉が変わる。

### 専門家 (査読者・指導教員)

特徴:
- 分野の文脈を理解している
- 技術的な厳密性を重視
- 既存研究との差別化を厳しく見る

強調すべき点:
- 関連研究の網羅性と差別化
- 理論的根拠と数式
- 実験設計の厳密性
- アブレーションでの寄与の切り分け

避けるべきこと:
- 過剰な背景説明(冗長に見える)
- 既存手法の単純化された説明(専門家が知っている)

### 助成機関・スポンサー

特徴:
- 必ずしも分野の専門家ではない
- インパクトとリソース効率を重視
- リスク管理の観点で見る

強調すべき点:
- インパクト(誰が、どう恩恵を受けるか)
- リスクと対応策
- マイルストーンと中間検証
- リソース計画の現実性

避けるべきこと:
- 過度に技術的な詳細
- 学術的な細部
- 専門用語の多用

### 自分自身

特徴:
- 研究の方向性を整理するため
- 後から読み返して進捗を確認する

強調すべき点:
- 仮説の明文化
- 成功基準の段階化
- 失敗時の代替案

書き方:
- 自分への正直さを最優先
- 「やりたいこと」と「実現可能なこと」を区別
- 不確実性を隠さない

### 一般の読者・科学コミュニケーション

特徴:
- ML 分野の前提知識がない
- 「なぜ重要か」を理解したい
- 比喩や具体例を求める

強調すべき点:
- 平易な要約
- 具体例
- 社会的意義

避けるべきこと:
- 専門用語
- 数式
- 内部ベンチマークの数字

ICML 2025 から「lay summary (平易な要約)」の提出が求められる動きがあり、この方向性が重視されている。

## チェックリスト

研究計画書を提出する前に確認すべき項目。

### 内容のチェック

- [ ] Heilmeier 質問 1-8 に明確に答えられているか
- [ ] 仮説は反証可能な形で書かれているか
- [ ] 数値目標が示されているか
- [ ] ベースラインが具体名で挙げられているか
- [ ] アブレーション計画があるか
- [ ] リスクと対応策が書かれているか
- [ ] ネガティブシナリオへの対応があるか
- [ ] 成功基準が段階化されているか
- [ ] マイルストーンに中間検証があるか
- [ ] リソース計画(人員、計算、時間)が現実的か

### 構成のチェック

- [ ] 平易な要約が冒頭にあるか
- [ ] 関連研究の差別化が明確か
- [ ] 技術詳細と動機のバランスが取れているか
- [ ] 図表が効果的に使われているか
- [ ] 引用が網羅的か(分野の代表的論文を含む)

### 表現のチェック

- [ ] 第 1 部冒頭に専門用語がないか
- [ ] 主張が定量的か(「向上する」ではなく「X% 改善」)
- [ ] 自分の研究の弱点を隠していないか
- [ ] 範囲が広すぎないか(1 つの主張に絞られているか)
- [ ] 受動態より能動態が多いか
- [ ] 文章が冗長になっていないか

### 戦略のチェック

- [ ] 投稿先(会議・ジャーナル)が決まっているか
- [ ] その投稿先の評価基準を満たすか
- [ ] 競合研究の動向を把握しているか
- [ ] 不足事項が明示されているか(ドラフト段階の場合)

### メタチェック

- [ ] 専門外の人に読んでもらって、第 1 部だけで研究の意義が伝わるか
- [ ] 専門家に読んでもらって、第 2 部で技術的に納得してもらえるか
- [ ] 自分が査読者だったら、どこを批判するか? それに答えられるか?

## 参考: Heilmeier 教義の起源

George H. Heilmeier (1936-2014) は液晶ディスプレイ (LCD) の発明者であり、1975-1977 年に DARPA のディレクターを務めた。彼が DARPA で導入した提案評価の質問群が「Heilmeier Catechism」として知られ、現在も国防省、NSF、NIH などの研究助成機関で評価軸の一部として使われている。

Heilmeier 教義の根底にある思想は「**興味深い研究の全てが重要なわけではない**」というもの。研究の知的好奇心と、社会的・実用的な価値を分けて評価し、両方を満たすものに資源を配分する哲学である。

## 補足: 文書の長さ目安

| 用途 | ページ数 | 主要セクション |
|---|---|---|
| 社内提案・予備検討 | 2-3 ページ | 動機、提案、検証計画(簡略) |
| グラント申請 | 5-15 ページ | フル構成、リスク管理を厚く |
| 学位論文計画書 | 10-30 ページ | フル構成、文献レビューを厚く |
| 一枚要約 (one-pager) | 1 ページ | Heilmeier 質問の凝縮版 |
| 査読論文の Method 提案部分 | 1-2 セクション | 技術的内容のみ抽出 |

長さは「読み手が必要としている情報量」で決まる。短ければ良い、長ければ良いではない。

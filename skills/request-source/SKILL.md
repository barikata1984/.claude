## 核心原則

応答中の参照情報は **デフォルトで [未検証]**. 当該セッション中に WebSearch / WebFetch / MCP で実際に確認したものだけを [検証済] に昇格する. 確認していない参照情報を Sources ブロックから省略してはならない.

応答を書き終えたら, 本文中の各事実主張を走査し, 「この事実をこのセッションで外部確認したか?」を判定する. 確認していない参照情報が 1 つでもあれば [未検証] として必ず列挙する.

## 動作モード

1. **自動付与**: 応答本文の末尾に Sources ブロックを追記する. 応答本文と Sources ブロックの間に `---` を置く.
2. **手動呼び出し** (`$ARGUMENTS` あり): 指定された応答(「直前の回答」等)を分析し, Sources ブロックのみを出力する. 冒頭の `---` は不要(本文がないため分離の必要がない). `Sources:` から始める.

## 出力フォーマット

```
---
Sources:
- [検証済] 項目の要約 — 参照先 (URL/DOI/ドキュメントパス)
- [未検証] 項目の要約(内部知識)
```

## 分類ルール

- **[検証済]**: 当該セッション中に WebSearch / WebFetch / MCP で実際に確認した情報. 参照先(URL, DOI, ドキュメントパス等)を必ず付記する.
- **[未検証]**: 訓練データ(内部知識)に依拠しており, 当該セッション中に外部検証していない情報. 参照先は書かない. 「(内部知識)」と付記する.
- **優先順位**: 粒度ルール(自明な常識 → 除外)は分類ルールに優先する. ただし除外は下記「除外基準」に該当するものに限定し, 迷ったら含める(除外の過誤は冗長の過誤より重大).

## 記載の粒度

- 応答中の主要な事実主張ごとに 1 行. 同一ソースから得た密接に関連する事実(例: 1 つの API の複数パラメータ)は 1 行にまとめてよい. 応答の構造的要素(「以下に示す」等)は含めない.
- **除外基準**(以下に該当するもののみ Sources から除外してよい):
  - 特定プロダクトに依存しない分野共通の基本概念(例: 「HTTP はステートレス」「勾配降下法で学習」「JavaScript はシングルスレッド」). 特定のライブラリ・フレームワーク・ツールの特性は, たとえ広く知られていても除外しない
- **含める**(以下に該当するものは自明に見えても Sources に記載する):
  - 特定のプロダクト・バージョン・API に関する事実
  - 「一般的」「標準的」「推奨」等の慣行判断を含む主張
  - 数値・日付・リリース情報を伴う主張
- 同一の参照先が複数項目に対応する場合, 各項目に同じ参照先を繰り返してよい.
- 同一の項目が検証済と未検証の両方に該当する場合(部分的に検証した場合), 検証済の行に記載し, 未検証部分があればその旨を補記する.

## CLAUDE.md の inline 注記との関係

CLAUDE.md「外部ソース事前参照」は本文中に「内部知識・未確認」と明示するルールを定めている.

- Sources ブロックを**付与する**応答: 未検証情報の明示は Sources ブロックの `[未検証]` に集約し, 本文中の個別注記(`(内部知識・未確認)` 等)は省略する(二重記載を避ける).
- Sources ブロックを**付与しない**応答(付与判断の基準を満たさない場合): CLAUDE.md のルールに従い, 本文中に「内部知識・未確認」を明示する.

どちらか一方は必ず適用される. 参照情報に対して両方とも適用しない状態は許されない.

## 付与判断の基準

応答に以下のいずれかが含まれるとき付与する:
- ライブラリ・フレームワーク・API の仕様やバージョン情報
- 論文の著者, 年, 数値, 手法の詳細
- 統計データ, ベンチマーク結果
- ニュース, イベント, リリース情報
- 第三者ドキュメントの内容への言及

以下のみで構成される応答には付与しない:
- コード編集・生成の結果報告
- ファイル・git 操作の実行報告
- セッション内の観測情報(エラーログ, テスト出力等)の報告
- 会話的やりとり(確認, 質問, 提案)
- ユーザーのコードベース内の事実のみに基づく説明

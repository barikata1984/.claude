# ロボット学習プロジェクト: フェーズ別作業分担表

## チームメンバー一覧

### 人間チーム

| 略称 | 正式名称 | 担当者 | 役割 |
| ---- | -------- | ------ | ---- |
| PM/PI | Project Manager / Principal Investigator | 人間 | プロジェクト全体の意思決定・方向性判断。学術モードでは研究責任者(PI)、スタートアップモードではプロダクトマネージャー(PM)を兼ねる |
| 研究者 | Researcher | 人間 | 技術的な判断・実装方針の決定・結果の解釈。PM/PI と並ぶコアメンバーで、実験の設計・考察に深く関与する |
| Oper. | Robot Operator | 人間 | 実機ロボットの操作・テレオペレーション・物理データ収集。Phase 3-4 で必要となる。エージェントでは代替不可能な身体的作業を担当 |

### エージェントチーム

| 略称 | 正式名称 | 担当者 | 役割 |
| ---- | -------- | ------ | ---- |
| Literature | Literature / Market Research Agent | LLMエージェント | 論文検索・要約・分類（学術モード）、競合調査・特許検索・市場分析（スタートアップモード） |
| Code | Code Implementation Agent | LLMエージェント | アルゴリズム実装・ベースライン再現・テスト作成・リファクタリング。コードの生成と修正を担当 |
| Experiment | Experiment Design & Analysis Agent | LLMエージェント | 実験マトリクス設計・統計検定計画・結果の統計処理・図表生成・効果量算出 |
| Writing | Writing & Communication Agent | LLMエージェント | 論文草稿・グラント申請書（学術モード）、技術ドキュメント・投資家向け資料（スタートアップモード） |
| Review | Review & Quality Gate Agent | LLMエージェント | コードレビュー・論文レビュー・統計手法の妥当性チェック。PASS/REVISE/BLOCK の3段階で品質判定 |
| Infra | Infrastructure & MLOps Agent | LLMエージェント | CI/CD構築・Docker環境・wandb設定・GPUジョブ管理・デプロイ準備 |

## ループ構造の一覧

| ループ | 経路 | トリガー | 想定回数 |
| ------ | ---- | -------- | -------- |
| (A) 仮説修正 | Ph.5 → Ph.2 → Ph.3 → Ph.4 → Ph.5 | 実験結果が仮説を否定 | 1-3回 |
| (B) 追加実験 | Ph.5 → Ph.4 → Ph.5 | 結果が不明瞭（有意差なし、サンプル不足） | 1-2回 |
| (C) 重大欠陥 | Ph.7 → Ph.3/4 → Ph.5 → Ph.6 → Ph.7 | Review が BLOCK 判定（再現不能、重大バグ） | 0-1回 |
| (D) 軽微修正 | Ph.7 内で完結 | Review が REVISE 判定 | 1-3回 |

> 注: (A) が3回を超える場合は Phase 0 の問題設定自体の見直しを検討。(C) が2回以上は設計の根本問題を示唆。

## 各フェーズの詳細

### Phase 0: 着想・問題定義

| 項目 | 内容 |
|------|------|
| 目的 | 解くべき問題を明確化し「何が面白いか/何に価値があるか」を言語化する |
| 主担当 | PM/PI（意思決定）、研究者（技術的実現可能性の判断） |
| エージェント | **不使用**。問題設定の質がプロジェクト全体の価値を決定するため、人間の直感と判断が不可欠 |
| 成果物 | 問題定義文書（1ページ以内） |
| 次フェーズへの条件 | 「何を解くか」「なぜそれが重要か」が1文で説明できる状態 |

### Phase 1: 先行研究 / 市場調査

| 項目 | 内容 |
|------|------|
| 目的 | 問題の既知/未知の境界を特定する |
| 主担当 | Literature Agent |
| 人間の役割 | 調査結果の評価、差別化ポイントの判断 |
| 学術モード | 論文網羅・研究ギャップ特定。`/literature-survey` に委譲可 |
| startup モード | 競合調査・特許検索・市場レポート分析 |
| 成果物 | 先行研究/競合分析レポート |
| 次フェーズへの条件 | 「既存研究/製品との差分」が明確に言語化できている |

### Phase 2: 仮説構築 / 実験設計

| 項目 | 内容 |
|------|------|
| 目的 | 検証可能な仮説を立て、実験計画を策定する |
| 主担当 | PM/PI（仮説策定）、Experiment Agent（実験設計） |
| 支援 | Literature Agent（既存ベースラインの評価条件収集） |
| 成果物 | 実験計画書（変数、条件、指標、統計検定計画、コスト見積） |
| 承認ゲート | PM/PI が実験計画を承認（コスト・期間のトレードオフ判断） |

### Phase 3: 実装 / プロトタイプ

| 項目 | 内容 |
|------|------|
| 目的 | 仮説検証に必要なコードとインフラを構築する |
| 主担当 | Code Agent（実装）、Infra Agent（環境構築）— 並行実行可 |
| 品質ゲート | Review Agent によるコードレビュー |
| 人間の役割 | 実機統合（ロボットオペレーター）— エージェント不可 |
| 成果物 | 実行可能なコード + テスト + 実験環境 |

### Phase 4: 実験・検証

| 項目 | 内容 |
|------|------|
| 目的 | 実験計画に従いデータを収集する |
| 主担当 | Oper.（実機実験）、Infra Agent（ジョブ管理） |
| エージェント活用 | 限定的。実験の実行自体は計算資源と物理ロボットの問題 |
| 既存スキル連携 | `/sweep run` でハイパーパラメータ探索を自動化 |
| 成果物 | 実験データ（wandb ログ、CSV、動画等） |

### Phase 5: 分析・考察

| 項目 | 内容 |
|------|------|
| 目的 | 実験データから結論を引き出す |
| 主担当 | Experiment Agent（統計処理）、PM/PI（解釈・考察） |
| 品質チェック | Review Agent（統計手法の妥当性確認） |
| 既存スキル連携 | `/sweep analyze` で結果分析を自動化 |
| 判定分岐 | 仮説支持→Phase 6 / 仮説否定→Phase 2 / 不明瞭→Phase 4 |
| 成果物 | 分析レポート（統計表、図、考察） |

### Phase 6: 出力（論文 or 製品）

| 項目 | 内容 |
|------|------|
| 目的 | 成果物を形にする |
| 主担当 | Writing Agent |
| 支援 | Literature Agent（引用補完）、Code Agent（再現コード/プロダクションコード） |
| 人間の役割 | ストーリー構成判断、主張の最終決定 |
| 学術モード | 論文草稿（Introduction〜Conclusion） |
| startup モード | 技術ドキュメント、投資家向け資料、デプロイ準備 |
| 成果物 | 論文ドラフト or リリース可能なプロダクト |

### Phase 7: レビュー / 改善ループ

| 項目 | 内容 |
|------|------|
| 目的 | 外部の目で成果物の質を担保する |
| 主担当 | Review Agent（判定）→ 各 Agent が修正対応 |
| 判定 | PASS（Phase 8 へ）/ REVISE（修正→再レビュー）/ BLOCK（Phase 3-4 へ差し戻し） |
| 人間の役割 | レビュー結果の優先度判断、対応方針決定 |
| 成果物 | 品質保証済みの最終成果物 |

### Phase 8: 公開・展開

| 項目 | 内容 |
|------|------|
| 目的 | 成果を世に出す |
| 主担当 | PM/PI（意思決定・実行） |
| 支援 | Infra Agent（リポジトリ公開準備、デプロイ） |
| エージェント活用 | 最小限。公開の意思決定は人間が行う |
| 成果物 | 投稿済み論文 or リリース済み製品 |
